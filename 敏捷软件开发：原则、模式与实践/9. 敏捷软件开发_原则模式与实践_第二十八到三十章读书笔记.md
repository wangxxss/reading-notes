## 第二十八章：VISITOR模式

### 一、模式定义

> "表示一个作用于某对象结构中的各元素的操作。访问者模式允许你在不改变各元素的类的前提下，定义作用于这些元素的新操作。"

### 二、解决的问题

当数据结构稳定，但需要频繁添加新操作时：

```java
// 数据结构（Element）
public interface Modem {
    void dial(String number);
    void hangup();
    void send(char c);
    char recv();
}

// 问题：如果要添加新操作（如配置），需要修改所有Modem类
// 违反OCP原则
```

### 三、UML结构

```
┌─────────────────┐
│    Visitor      │    ┌─────────────────┐
│ +visit(Modem)   │───▶│      Modem      │
└─────────────────┘    │ +accept(Visitor)│
        △              └─────────────────┘
        │                       △
┌───────┴───────┐                │
│               │         ┌──────┴──────┐
│   UnixVisitor │         │              │
└───────────────┘   ┌─────┴─────┐  ┌────┴────┐
│   DosVisitor  │   │  UnixModem│  │DosModem │
└───────────────┘   └───────────┘  └─────────┘
```

### 四、代码示例

```java
// 访问者接口
public interface ModemVisitor {
    void visit(UnixModem m);
    void visit(DosModem m);
}

// 元素接口
public interface Modem {
    void accept(ModemVisitor v);
}

// 具体元素
public class UnixModem implements Modem {
    public void accept(ModemVisitor v) {
        v.visit(this);
    }
}

// 具体访问者
public class UnixConfigVisitor implements ModemVisitor {
    public void visit(UnixModem m) {
        m.initializeForUnix();
    }
    public void visit(DosModem m) {
        // DOS配置
    }
}
```

### 五、双重分发（Double Dispatch）

```java
// 第一次分发：确定元素的类型
modem.accept(visitor);  // 调用的是UnixModem的accept

// 第二次分发：调用visitor的visit方法
visitor.visit(this);  // 调用的是visit(UnixModem)
```

### 六、VISITOR模式的代价

#### 优点
- 符合OCP：增加新操作不需修改元素类
- 集中相关操作在一起
- 访问者可以累积状态

#### 缺点
- 增加新元素困难：需要修改所有Visitor
- 打破了封装：Visitor需要了解元素内部结构

### 七、适用场景

- 数据结构相对稳定
- 需要经常添加新操作
- 适合树形结构遍历

## 第二十九章：STATE模式

### 一、模式定义

> "允许对象在内部状态改变时改变它的行为。对象看起来好像修改了它的类。"

### 二、解决的问题

```java
// 问题代码：大量的条件判断
public class TCPConnection {
    public void open() {
        if (state == ESTABLISHED) {
            // ...
        } else if (state == LISTEN) {
            // ...
        }
    }
}
```

### 三、UML结构

```
┌─────────────────┐
│     Context     │
│ - state         │
│ +request()      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│    State       │◀─────────────┐
│ +handle()      │              │
└────────┬────────┘              │
         │              ┌────────┴────────┐
    ┌────┴────┐        │                 │
    ▼         ▼        ▼                 ▼
┌──────┐ ┌──────┐ ┌────────┐      ┌────────┐
│StateA │ │StateB │ │StateC │      │StateD │
└───────┘ └──────┘ └────────┘      └────────┘
```

### 四、代码示例

```java
// 状态接口
public interface ConnectionState {
    void open(TCPConnection connection);
    void close(TCPConnection connection);
    void acknowledge(TCPConnection connection);
}

// 具体状态
public class Established implements ConnectionState {
    public void open(TCPConnection c) {
        throw new IllegalStateException();
    }
    public void close(TCPConnection c) {
        c.setState(new Closed());
    }
    public void acknowledge(TCPConnection c) {
        // 处理ACK
    }
}

// 上下文
public class TCPConnection {
    private ConnectionState state;

    public void setState(ConnectionState s) {
        this.state = s;
    }

    public void close() {
        state.close(this);
    }
}
```

### 五、STATE vs STRATEGY

| 对比 | STATE | STRATEGY |
|-----|-------|----------|
| 意图 | 对象状态变化导致行为变化 | 算法可以替换 |
| 状态关系 | 状态之间通常有转换关系 | 算法之间通常平等 |
| 上下文控制 | 状态控制转换 | 客户控制算法选择 |
| 数量 | 通常有限 | 可以很多 |

### 六、STATE模式的价值

- 消除大量if-else/switch
- 状态转换逻辑集中管理
- 符合OCP：添加新状态不需改上下文
- 每个状态类职责单一

## 第三十章：ETS框架（Expedition Tools Software）

### 一、案例背景

ETS是一个用于处理第三方API的框架，需要：
1. 与多个第三方服务交互
2. 统一的错误处理
3. 可扩展的API支持

### 二、框架设计原则

#### 1. 依赖倒置

```
┌─────────────────┐
│   Application   │  高层模块
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Abstract API  │  抽象层
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Concrete API  │  低层模块
└─────────────────┘
```

#### 2. 模板方法应用

```java
public abstract class APIHandler {
    // 模板方法
    public final void handle(Request request) {
        validate(request);
        preprocess(request);
        doHandle(request);
        postprocess(request);
    }

    protected abstract void doHandle(Request request);
}
```

#### 3. 策略模式应用

不同类型的请求使用不同的处理策略：

```java
public interface RequestStrategy {
    Response execute(Request request);
}

public class PriorityRequestStrategy implements RequestStrategy {
    public Response execute(Request request) {
        // 优先级处理
    }
}
```

### 三、设计模式综合应用

| 模式 | 应用位置 |
|-----|---------|
| Facade | 统一入口 |
| Adapter | 适配第三方API |
| Strategy | 请求处理策略 |
| Template Method | 处理流程骨架 |
| State | 请求状态管理 |
| Visitor | API配置操作 |

### 四、框架的关键洞见

> "框架定义了问题域的骨架，而应用程序则是填充骨架的血肉。"

> "好的框架应该：允许应用程序插入自己的策略；提供默认策略；易于扩展而非修改。"

### 五、框架vs库

| 对比 | 框架 | 库 |
|-----|------|-----|
| 控制流 | 框架控制 | 应用程序控制 |
| 扩展点 | 框架提供钩子 | 应用程序调用 |
| 设计 | Hollywood Principle | 不要调用我，我会调用你 |

## 第二十八至三十章核心总结

### 一、模式速查

| 模式 | 核心思想 | 解决的问题 |
|-----|---------|-----------|
| Visitor | 双重分派 | 数据结构稳定，需要添加新操作 |
| State | 状态转换对象化 | 大量条件判断，状态变化影响行为 |

### 二、模式选择指南

```
数据结构稳定，添加新操作？ → VISITOR
状态变化影响行为？ → STATE
需要替换算法？ → STRATEGY
控制流需要反转？ → TEMPLATE METHOD
```

### 三、设计原则应用

| 原则 | 应用 |
|-----|------|
| OCP | Visitor允许添加新操作不修改元素 |
| SRP | 每个State类职责单一 |
| DIP | 框架依赖抽象，应用程序实现 |

### 四、关键洞见

> "Visitor模式通过双重分派，实现了操作与数据结构的分离。"

> "State模式使得状态转换变得清晰，每个状态都知道下一步该去哪里。"

> "框架是冻结的策略 + 开放的扩展点。"

> "好的设计不是一蹴而就的，而是在迭代中逐渐演化的。"

## 全书模式总结

### 一、创建型模式

| 模式 | 目的 |
|-----|------|
| Factory | 将对象创建与使用分离 |
| Singleton | 控制实例数量 |
| Monostate | 强制行为单一性 |

### 二、结构型模式

| 模式 | 目的 |
|-----|------|
| Composite | 统一处理单个和组合对象 |
| Adapter | 接口转换 |
| Bridge | 分离抽象与实现 |
| Facade | 简化复杂子系统接口 |
| Proxy | 控制对象访问 |

### 三、行为型模式

| 模式 | 目的 |
|-----|------|
| Command | 封装请求为对象 |
| Observer | 一对多通知 |
| Strategy | 算法可替换 |
| State | 状态影响行为 |
| Visitor | 操作与数据结构分离 |
| Template Method | 算法骨架 + 钩子方法 |
| Mediator | 解耦对象交互 |

### 四、其他模式

| 模式 | 目的 |
|-----|------|
| Null Object | 替代null，提供默认行为 |
| Active Object | 异步执行，解耦调用和执行 |
