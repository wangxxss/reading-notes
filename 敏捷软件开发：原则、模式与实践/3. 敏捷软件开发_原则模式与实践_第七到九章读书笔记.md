# 《敏捷软件开发：原则、模式与实践》第七至九章读书笔记

---

## 第七章：什么是敏捷设计

### 一、敏捷设计的核心思想

**敏捷设计不是"预先设计"出来的，而是通过迭代演化出来的。**

敏捷团队不会在一开始设计模块时就试图预测程序会怎么变化，不进行预先设计。而是：
1. 使用单元测试和验收测试来辅助
2. 使系统设计尽可能干净、简单
3. 直到需求变化的时候才修改模块设计
4. 使模块对该种变化保持弹性

---

### 二、拙劣设计的症状（设计的臭味）

当软件设计开始腐化时，会出现以下症状：

#### 1. 僵化性（Rigidity）
- **定义**：设计难以改变
- **表现**：对软件进行一个单一改动，却引发一系列有依赖关系模块的连锁改动
- **原因**：模块之间耦合过紧

#### 2. 脆弱性（Fragility）
- **定义**：设计易于遭到破坏
- **表现**：对软件进行单一改动时，程序的许多其他地方会出现问题
- **危害**：一个小改动破坏多个地方

#### 3. 牢固性（Immobility）
- **定义**：设计难以重用
- **表现**：设计中包含了对其他系统有用的部分，但是把这部分从系统中分离出来代价和风险很大
- **问题**：好的东西无法复用

#### 4. 粘滞性（Viscosity）
- **定义**：难以做正确的事情
- **表现**：
  - 保持系统设计的改动方式比破坏设计的改动方式更难实现
  - 开发环境迟钝笨重导致开发人员不想编写需要大规模重编译的改动代码

#### 5. 不必要的复杂性（Needless Complexity）
- **定义**：过分设计
- **表现**：为过多的可能性做了准备，致使多了很多不会用到的设计结构，让程序变得不好理解
- **问题**：过度设计比设计不足更糟糕

#### 6. 不必要的重复（Needless Repetition）
- **定义**：滥用鼠标进行拷贝粘贴
- **表现**：复制粘贴并小改动代码，开发人员忽视了抽象
- **问题**：重复代码难以维护

#### 7. 晦涩性（Opacity）
- **定义**：混乱的表达
- **表现**：模块难以理解，开发人员没有站在代码阅读者的角度
- **解决**：需要重构代码

---

### 三、软件腐化的原因

> "软件项目中需求是持续变化的，如果软件设计由于需求变化而退化，那我们就不是敏捷的。"

**两大原因：**
1. **需求没有按照设计预见的方式进行变化**
2. **改动急迫且违反了原来的设计**

---

### 四、如何保持好的设计

**敏捷团队应该：**

1. **遵循敏捷实践去发现问题** - 通过测试和反馈
2. **应用设计原则去诊断问题** - 使用SOLID原则
3. **应用适当的设计模式去解决问题** - 合适的模式应用

> "最好的架构、需求和设计出自于自组织的团队。"

---

### 五、敏捷设计与传统设计的区别

| 方面 | 传统设计 | 敏捷设计 |
|-----|---------|---------|
| 时机 | 项目开始时做大量预设计 | 迭代中逐步演化 |
| 假设 | 试图预测所有可能的变化 | 等变化发生再应对 |
| 重构 | 后期大规模重构 | 持续小步重构 |
| 复杂度 | 过度设计以防万一 | 简单设计，按需演进 |

## 第八章：单一职责原则（SRP）

### 一、SRP定义

> **"一个类应该只有一个发生变化的原因。"**

在SRP中，**职责**被定义为**变化的原因**。

如果你能想到多于一个的修改动机，那这个类就具有多个职责。

---

### 二、为什么需要SRP

#### 多职责的问题

如果一个类有多个职责（多根轴线）：
- 引起类变化的原因就有多个
- 一个职责的变化可能会**抑制**该类完成其他职责的能力
- 职责之间会相互影响

#### 示例

```
违反SRP的类：Employee
├── 计算工资 (PayCalculation)
├── 输出考勤报表 (ReportGeneration)
├── 保存到数据库 (DatabaseOperations)
└── 发送邮件通知 (EmailNotification)

问题：如果数据库结构变化，整个Employee类都要改！
```

#### 符合SRP的设计

```
拆分为：
├── Employee (只有业务逻辑)
├── PayCalculator (工资计算)
├── ReportGenerator (报表生成)
├── EmployeeRepository (数据持久化)
└── EmailService (邮件服务)
```

---

### 三、识别违反SRP的信号

1. **类中的代码行数过多** - 可能承担了太多职责
2. **一个类的多个改动原因** - 需求变化总是同时影响多个功能
3. **测试难以编写** - 测试要覆盖太多不同的行为
4. **方法被频繁修改** - 不同的开发者在修改同一个类的不同方法

---

### 四、SRP的实际应用

#### 何时分离职责

> "常常会有一些原因迫使我们把不愿耦合在一起的东西耦合在一起。"

**建议策略**：
1. 对于没有遵守SRP的类，**只有在需求发生变化并导致职责轴线受到影响的时候**才去对该类执行职责分离
2. 如果需求的变化**总是导致两个职责同时变化**，就没有必要去分离

#### 重构方法

当需要分离职责时，可以使用以下模式：
- **Facade模式** - 为子系统提供一个简化的接口
- **Proxy模式** - 为另一个对象提供替身或占位符
- **拆分为多个小类** - 每个类只负责一件事

---

### 五、SRP的常见误区

#### 误区1：职责分得越细越好
**事实**：如果两个职责总是同时变化，分离它们反而会增加复杂性。

#### 误区2：一个类只能有一个方法
**事实**：SRP关注的是**变化的原因**，而不是方法的数量。

#### 误区3：测试驱动开发不需要SRP
**事实**：测试驱动的开发模式常常会使我们**很早就将职责分离**。

---

### 六、SRP与代码示例

#### 违反SRP的代码特征

```java
// 违反SRP：一个类有多个变化原因
class UserManager {
    void authenticate() { }        // 认证方式可能变化
    void updateProfile() { }       // 个人资料格式可能变化
    void saveToDatabase() { }     // 数据库结构可能变化
    void generateReport() { }      // 报表格式可能变化
    void sendEmail() { }          // 邮件服务可能变化
}
```

#### 符合SRP的设计

```java
// 符合SRP：每个类只有一个变化原因
class AuthenticationService {
    void authenticate() { }
}

class ProfileManager {
    void updateProfile() { }
}

class UserRepository {
    void save() { }
    void find() { }
}

class ReportGenerator {
    void generate() { }
}

class EmailService {
    void send() { }
}
```

## 第九章：开放-封闭原则（OCP）

### 一、OCP定义

> **"软件实体应该是可以扩展的，即开放的；但是不可修改的，即封闭的。"**

简称：**对扩展开放，对修改关闭**

---

### 二、OCP的核心特征

#### 1. 对扩展是开放的（Open for Extension）
- 模块的功能可以根据需求变化扩展
- 可以添加新的功能、新的行为
- 模块可以被继承、可以被扩展

#### 2. 对修改是封闭的（Closed for Modification）
- 扩展模块功能时，不改动源代码和二进制文件
- 原有代码不需要重新测试
- 不破坏已有的功能

---

### 三、OCP的关键：抽象

**如何做到"扩展却不修改"？**

答案是：**抽象**。

> "对模块的基础操作依赖于一个固定的抽象体，所以可以做到不更改模块基础操作的同时从这个抽象体派生来扩展模块的行为。"

#### 示例

```
                    ┌─────────────┐
                    │  Shape      │  ← 抽象基类（固定的）
                    │ (抽象封闭)  │
                    └─────────────┘
                           △
                           │
            ┌──────────────┴──────────────┐
            │                             │
     ┌──────┴──────┐              ┌───────┴───────┐
     │   Circle    │              │   Rectangle   │
     │ (扩展开放)  │              │  (扩展开放)   │
     └─────────────┘              └───────────────┘
```

- 添加新形状时，不需要修改 `AreaCalculator` 类
- 只需要继承 `Shape` 创建新的子类

---

### 四、OCP的实践策略

#### 1. 预测变化 vs 等待变化

> "由于遵循OCP要花很多时间精力去创建正确的抽象，所以在有限的成本下，把OCP尽可能应用在可能发生的变化，并且**直到发生变化时才采取行动**。"

**策略**：
- **不要**过度抽象
- **不要**为所有可能的变化做准备
- **直到**真的发生变化时才创建抽象

#### 2. 隔离变化的技术

> "在我们认为可能发生变化的地方放置钩子（hook）并维护它。"

- 识别可能变化的部分
- 为这些部分创建稳定的抽象接口
- 通过继承扩展，而不是修改已有代码

#### 3. 代价平衡

> "只承担一次变化造成的改动代价，但不允许下一次变化带来更多的代价。"

---

### 五、OCP的代码示例

#### 违反OCP：使用if-else判断类型

```java
// 违反OCP：每添加一个新类型都要修改这个方法
class AreaCalculator {
    double calculateArea(Object shape) {
        if (shape instanceof Circle) {
            Circle c = (Circle) shape;
            return Math.PI * c.getRadius() * c.getRadius();
        } else if (shape instanceof Rectangle) {
            Rectangle r = (Rectangle) shape;
            return r.getWidth() * r.getHeight();
        }
        // 添加新形状？修改这个方法！
        return 0;
    }
}
```

#### 符合OCP：使用多态

```java
// 符合OCP：添加新形状不需要修改这个方法
abstract class Shape {
    abstract double area();
}

class Circle extends Shape {
    double radius;
    double area() {
        return Math.PI * radius * radius;
    }
}

class Rectangle extends Shape {
    double width, height;
    double area() {
        return width * height;
    }
}

class AreaCalculator {
    double calculateArea(Shape shape) {
        return shape.area();  // 封闭的，不需要修改
    }
}
```

### 六、OCP与其他原则的关系

#### OCP是SOLID原则的核心

> "OCP是OOD中很多说法的核心。如果这个原则应用得有效，应用程序就会具有更多的可维护性、可重用性以及健壮性。"

**SOLID原则中其他原则都是为了支持OCP**：
- **SRP**：通过分离职责，使变化局部化
- **LSP**：保证子类型可以替换基类型
- **DIP**：通过依赖抽象来实现扩展
- **ISP**：通过分离接口来隔离变化

---

### 七、OCP的常见误区

#### 误区1：必须预测所有可能的变化
**事实**：不需要为所有可能的变化做准备。应该在变化发生时再进行抽象。

#### 误区2：任何代码都不能修改
**事实**：代码在必要时可以修改，但要确保修改后仍然符合OCP。

#### 误区3：过度抽象
**事实**：过度抽象会增加复杂性，应该在可维护性和复杂性之间找到平衡。

## 第七至九章核心总结

### 一、敏捷设计的核心

```
┌─────────────────────────────────────────────────────────┐
│                      敏捷设计                            │
├─────────────────────────────────────────────────────────┤
│  1. 识别腐化 → 2. 应用原则 → 3. 使用模式                  │
│  (发现症状)      (SOLID)         (解决方案)              │
└─────────────────────────────────────────────────────────┘
```

### 二、三种设计腐化症状速查

| 症状 | 问题 | 解决原则 |
|-----|------|---------|
| 僵化性 | 改动引发连锁反应 | SRP |
| 脆弱性 | 小改动大破坏 | OCP |
| 牢固性 | 好东西难复用 | DIP |

### 三、 SOLID 五大原则

| 原则 | 全称 | 核心思想 |
|-----|------|---------|
| **S**RP | 单一职责原则 | 一个类只有一个变化原因 |
| **O**CP | 开放-封闭原则 | 对扩展开放，对修改封闭 |
| **L**SP | 里氏替换原则 | 子类可以替换基类 |
| **I**SP | 接口隔离原则 | 接口要小而专 |
| **D**IP | 依赖倒置原则 | 依赖抽象，不依赖具体 |

### 四、设计原则的关系

```
                    SRP
                     △
                     │
              LSP ←──┼──→ OCP
                     │
                     │
                    DIP
```

- **SRP** 和 **OCP** 是其他原则的基础
- **LSP** 是 **OCP** 得以实现的主要手段
- **DIP** 告诉我们如何依赖抽象

### 五、关键洞见

> **"软件项目中需求是持续变化的，如果软件设计由于需求变化而退化，那我们就不是敏捷的。"**

> **"在软件的生命周期内，因为变化、升级和维护等原因需要对软件原有代码进行修改时，可能会给旧代码中引入错误。"**

> **"由于遵循OCP要花很多时间精力去创建正确的抽象，所以在有限的成本下，把OCP尽可能应用在可能发生的变化，并且直到发生变化时才采取行动。"**

> **"只承担一次变化造成的改动代价，但不允许下一次变化带来更多的代价。"**

## 附录：SOLID原则速记口诀

```
S - 单一职责：一个类做一件事
O - 开放封闭：扩展不难，修改不行
L - 里氏替换：子类能顶替父类
I - 接口隔离：接口要小而专
D - 依赖倒置：依赖抽象不依赖具体
```
