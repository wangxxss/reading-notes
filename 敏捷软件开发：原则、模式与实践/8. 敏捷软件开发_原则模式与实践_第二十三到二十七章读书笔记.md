## 第二十三章：COMPOSITE模式

### 一、模式定义

> "将对象组合成树形结构以表示'部分-整体'的层次结构。组合模式使得客户端对单个对象和组合对象的使用具有一致性。"

### 二、解决的问题

在没有组合模式的情况下：

```java
// 客户端需要区分叶子还是容器
if (component.isLeaf()) {
    component.process();
} else {
    for (child in component.getChildren()) {
        child.process();
    }
}
```

使用组合模式后：

```java
// 统一接口，客户端无需区分
component.process();
```

### 三、UML结构

```
        ┌─────────────┐
        │  Component  │
        │ +process() │
        │ +add()     │
        │ +remove()  │
        └──────┬──────┘
               │
       ┌───────┴───────┐
       │               │
┌──────┴──────┐  ┌────┴─────────┐
│    Leaf     │  │   Composite  │
│ +process()  │  │ +process()   │
│             │  │ -children[] │
└─────────────┘  └──────────────┘
```

### 四、代码示例

```java
// 抽象组件
public abstract class FileSystemComponent {
    public abstract void display();
    public void add(FileSystemComponent c) {
        throw new UnsupportedOperationException();
    }
    public void remove(FileSystemComponent c) {
        throw new UnsupportedOperationException();
    }
}

// 叶子节点
public class File extends FileSystemComponent {
    private String name;
    public void display() {
        System.out.println("File: " + name);
    }
}

// 组合节点
public class Folder extends FileSystemComponent {
    private List<FileSystemComponent> children = new ArrayList<>();

    public void add(FileSystemComponent c) {
        children.add(c);
    }

    public void display() {
        for (FileSystemComponent child : children) {
            child.display();
        }
    }
}
```

### 五、透明组合 vs 安全组合

| 类型 | 特点 |
|-----|------|
| **透明组合** | Component包含所有方法，Leaf也要实现add/remove（但可能抛异常） |
| **安全组合** | add/remove只在Composite中定义，Client需要类型判断 |

### 六、应用场景

- 文件系统（文件/文件夹）
- 组织架构（员工/部门）
- 菜单系统（菜单项/子菜单）
- 图形编辑器（图形/组合图形）

## 第二十四章：OBSERVER模式

### 一、模式定义

> "定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖它的对象都会收到通知并自动更新。"

### 二、核心角色

| 角色 | 职责 |
|-----|------|
| Subject（被观察者） | 维护观察者列表，状态变化时通知 |
| Observer（观察者） | 定义更新接口 |
| ConcreteSubject | 具体被观察者 |
| ConcreteObserver | 具体观察者 |

### 三、UML结构

```
┌─────────────────┐         ┌─────────────────┐
│    Subject      │         │    Observer     │
│ +attach()      │────────▶│ +update()      │
│ +detach()      │         └─────────────────┘
│ +notify()      │                   △
└────────┬───────┘                    │
         │                            │
    ┌────┴────┐              ┌───────┴───────┐
    │         │              │               │
┌───┴───┐ ┌──┴──────────┐    │  ConcreteObserver
│       │ │ConcreteSubject│    │
└───────┘ └─────────────┘    └───────────────┘
```

### 四、两种模型对比

#### 拉模型（Pull Model）

```java
// Subject
public void notify() {
    for (Observer o : observers) {
        o.update(this);  // 观察者自己来拉数据
    }
}

// Observer
public void update(Subject subject) {
    // 观察者主动从Subject获取需要的数据
    String data = subject.getData();
}
```

#### 推模型（Push Model）

```java
// Subject
public void notify() {
    for (Observer o : observers) {
        o.update(data);  // 直接把数据推给观察者
    }
}
```

### 五、OBSERVER模式的推动力

> "OBSERVER模式的最大推动力来自开放封闭原则（OCP）。使用这个模式的动机就是为了在增加新的观察对象时可以无需更改被观察的对象。"

### 六、Java内置支持

```java
// java.util.Observable (已过时)
public class Observable {
    public void addObserver(Observer o);
    public void deleteObserver(Observer o);
    public void notifyObservers();
}

// java.util.Observer (已过时)
public interface Observer {
    void update(Observable o, Object arg);
}
```

现代推荐使用：
- Java 9+ 的 Flow API (Reactive Streams)
- RxJava
- EventBus

## 第二十五章：ABSTRACT SERVER、ADAPTER和BRIDGE模式

### 一、ABSTRACT SERVER模式

#### 模式定义

> "为服务器创建一个抽象接口，使得客户端可以通过这个接口与服务器通信，而不需要知道服务器的具体类型。"

#### 应用场景

当系统需要支持多种数据库或多种服务时：

```java
// 抽象服务器接口
public interface DataServer {
    QueryResult executeQuery(String sql);
    void executeUpdate(String sql);
}

// MySQL实现
public class MySQLServer implements DataServer {
    public QueryResult executeQuery(String sql) { /* MySQL实现 */ }
    public void executeUpdate(String sql) { /* MySQL实现 */ }
}

// Oracle实现
public class OracleServer implements DataServer {
    public QueryResult executeQuery(String sql) { /* Oracle实现 */ }
    public void executeUpdate(String sql) { /* Oracle实现 */ }
}
```

### 二、ADAPTER模式（适配器）

#### 模式定义

> "将一个类的接口转换成客户希望的另一个接口。适配器模式使得原本由于接口不兼容而不能一起工作的类可以一起工作。"

#### 场景

```java
// 第三方库的接口
public class ThirdPartyLib {
    public void specialRequest() { /* ... */ }
}

// 客户端期望的接口
public interface Target {
    void request();
}

// 适配器
public class Adapter implements Target {
    private ThirdPartyLib adaptee;

    public void request() {
        adaptee.specialRequest();
    }
}
```

#### 类型

| 类型 | 说明 |
|-----|------|
| 类适配器 | 通过继承实现（Adapter继承Adaptee） |
| 对象适配器 | 通过组合实现（Adapter持有Adaptee引用） |

### 三、BRIDGE模式（桥接）

#### 模式定义

> "将抽象部分与它的实现部分分离，使它们都可以独立地变化。"

#### 场景

```java
// 抽象部分
public abstract class Shape {
    protected Renderer renderer;  // 桥接到实现

    public Shape(Renderer renderer) {
        this.renderer = renderer;
    }

    public abstract void draw();
}

// 实现部分
public interface Renderer {
    void renderCircle();
    void renderSquare();
}

// 具体实现
public class VectorRenderer implements Renderer {
    public void renderCircle() { /* 矢量绘制 */ }
    public void renderSquare() { /* 矢量绘制 */ }
}

public class RasterRenderer implements Renderer {
    public void renderCircle() { /* 光栅绘制 */ }
    public void renderSquare() { /* 光栅绘制 */ }
}

// 抽象部分的具体类
public class Circle extends Shape {
    public Circle(Renderer renderer) { super(renderer); }

    public void draw() {
        renderer.renderCircle();
    }
}
```

### 四、三种模式对比

| 模式 | 目的 | 分离的是什么 |
|-----|------|-------------|
| Abstract Server | 隔离客户端和服务器 | 客户端与具体服务器 |
| Adapter | 转换接口 | 现有接口与期望接口 |
| Bridge | 分离抽象与实现 | 抽象与实现平台 |

## 第二十六章：PROXY模式

### 一、模式定义

> "为另一个对象提供一个替身或占位符，以控制对这个对象的访问。"

### 二、为什么要用代理

| 场景 | 原因 |
|-----|------|
| 远程代理 | 访问远程对象，本地代表 |
| 虚代理 | 延迟加载 expensive 对象 |
| 保护代理 | 控制原始对象的访问权限 |
| 智能引用 | 当访问对象时执行额外操作 |

### 三、代码示例

```java
// 真实对象
public class RealImage {
    private String filename;

    public RealImage(String filename) {
        this.filename = filename;
        loadFromDisk();
    }

    public void display() {
        System.out.println("Displaying " + filename);
    }

    private void loadFromDisk() {
        System.out.println("Loading " + filename);
    }
}

// 虚代理
public class ImageProxy {
    private RealImage realImage;
    private String filename;

    public ImageProxy(String filename) {
        this.filename = filename;
    }

    public void display() {
        if (realImage == null) {
            realImage = new RealImage(filename);  // 延迟加载
        }
        realImage.display();
    }
}
```

### 四、代理模式vs装饰器模式

| 对比 | 代理模式 | 装饰器模式 |
|-----|---------|-----------|
| 目的 | 控制访问 | 添加职责 |
| 客户端感知 | 客户端不知道是代理 | 客户端不知道是装饰 |
| 接口 | 通常一样 | 通常一样 |
| 实现 | 通常持有真实对象引用 | 持有同一接口的引用 |

## 第二十七章：气象站案例研究

### 一、案例背景

气象站需要：
1. 收集温度、湿度、气压数据
2. 通知多个显示设备
3. 支持历史数据查询
4. 支持第三方设备接入

### 二、设计分析

#### 观察者模式应用

```java
// 气象站（Subject）
public class WeatherStation {
    private List<Display> displays = new ArrayList<>();
    private float temperature;
    private float humidity;
    private float pressure;

    public void registerDisplay(Display d) {
        displays.add(d);
    }

    public void notifyDisplays() {
        for (Display d : displays) {
            d.update(temperature, humidity, pressure);
        }
    }
}

// 显示设备（Observer）
public interface Display {
    void update(float temp, float humidity, float pressure);
}
```

#### 组合模式应用

气象数据本身可能需要组合：

```java
// 单一传感器
public class Sensor implements DataSource {
    private float value;
    public float getValue() { return value; }
}

// 传感器组
public class SensorGroup implements DataSource {
    private List<DataSource> sensors;
    public float getAverage() {
        // 计算所有传感器的平均值
    }
}
```

### 三、模式综合应用

| 模式 | 应用场景 |
|-----|---------|
| Observer | 气象数据变化通知显示设备 |
| Composite | 传感器组合管理 |
| Strategy | 不同传感器使用不同算法 |
| Factory | 创建不同类型的传感器 |

## 第二十三至二十七章核心总结

### 一、模式速查

| 模式 | 核心思想 | 解决的问题 |
|-----|---------|-----------|
| Composite | 树形结构统一接口 | 客户端需区分叶子/容器 |
| Observer | 一对多依赖 | 对象状态变化需通知多方 |
| Abstract Server | 抽象服务器接口 | 隔离客户端与具体服务器 |
| Adapter | 接口转换 | 接口不兼容 |
| Bridge | 分离抽象与实现 | 多维度变化 |
| Proxy | 控制访问 | 延迟加载/远程访问/访问控制 |

### 二、模式选择指南

```
需要表示树形结构？ → Composite
需要一对多通知？ → Observer
需要控制对象访问？ → Proxy
需要接口转换？ → Adapter
需要分离抽象与实现？ → Bridge
需要抽象服务器？ → Abstract Server
```

### 三、关键洞见

> "组合模式使得客户端可以统一处理单个对象和组合对象，无需关心它们的差异。"

> "OBSERVER模式的最大推动力来自开放封闭原则（OCP）。"

> "Bridge模式将抽象与实现分离，使得它们可以独立变化。"

> "代理模式在不改变原始对象的情况下，控制对它的访问。"
