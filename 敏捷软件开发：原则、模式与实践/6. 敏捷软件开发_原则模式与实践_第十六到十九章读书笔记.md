## 第十六章：SINGLETON模式和MONOSTATE模式

### 一、SINGLETON模式（单例模式）

#### 模式定义

确保一个类只有一个实例，并提供一个全局访问点。

#### 实现方式

```java
public class Singleton {
    private static Singleton instance;
    private Singleton() { }  // 私有构造函数

    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

#### 关键特征

- **私有构造函数**：防止外部直接创建实例
- **静态变量**：保存唯一实例
- **静态方法**：提供全局访问点

---

### 二、MONOSTATE模式

#### 模式定义

> "Monostate模式不强制结构的单一性，而是强制行为的单一性。"

#### 实现方式

```java
public class Monostate {
    private static int itsValue;  // 所有实例共享同一个静态变量

    public int getValue() { return itsValue; }
    public void setValue(int v) { itsValue = v; }
}

// 使用时
Monostate m1 = new Monostate();
Monostate m2 = new Monostate();
m1.setValue(10);  // 实际上修改的是同一个变量
System.out.println(m2.getValue());  // 输出10
```

#### 关键特征

- **所有实例共享同一组成员变量**（通过static实现）
- 客户端不知道使用的是Monostate
- 可以有多个实例，但状态是共享的
- **可以有派生类**，继承模式仍然有效

---

### 三、SINGLETON vs MONOSTATE

| 对比 | SINGLETON | MONOSTATE |
|-----|-----------|-----------|
| 实例 | 只有一个实例 | 可以有多个实例 |
| 状态 | 状态在单实例中 | 状态在静态变量中 |
| 继承 | 不能继承（构造函数私有） | 可以继承 |
| 客户端感知 | 客户端知道使用的是单例 | 客户端感知不到差异 |
| 创建方式 | 延迟创建或立即创建 | 任何构造函数 |

---

### 四、何时使用哪个

#### 选择SINGLETON的场景
- 需要严格控制实例数量
- 需要延迟初始化
- 不需要继承

#### 选择MONOSTATE的场景
- 希望多态使用
- 需要支持继承
- 希望客户端无感知

## 第十七章：NULL OBJECT模式

### 一、模式定义

> "用一个对象替代null，该对象实现了与真实对象相同的接口，但方法是空的或返回默认值。"

### 二、解决的问题

#### null引用的危害

```java
// 传统方式：大量null检查
if (user != null) {
    if (user.getAccount() != null) {
        if (user.getAccount().getBalance() != null) {
            // 处理业务
        }
    }
}
```

#### NULL OBJECT的解决方案

```java
// 定义用户接口
public interface User {
    String getName();
    boolean isNull();  // 标识是否为Null对象
}

// 真实用户
public class RealUser implements User {
    private String name;
    public String getName() { return name; }
    public boolean isNull() { return false; }
}

// Null对象
public class NullUser implements User {
    public static final NullUser INSTANCE = new NullUser();
    public String getName() { return "Anonymous"; }
    public boolean isNull() { return true; }
}
```

### 三、空对象模式的价值

| 好处 | 说明 |
|-----|------|
| **消除null检查** | 代码不再充斥着 if(obj == null) |
| **统一行为** | 所有对象都实现了相同接口 |
| **可预测性** | NullObject方法是空的，没有副作用 |
| **符合LSP** | NullObject可以替换真实对象 |

### 四、应用场景

- 用户未登录 → 返回NullUser
- 找不到数据 → 返回NullResult
- 没有权限 → 返回NullPermission

## 第十八章：薪水支付案例研究 - 第一次迭代开始

### 一、案例背景

薪水支付系统需要：
1. 为每个雇员支付薪水
2. 按照规定方法准时支付正确数目
3. 从薪水中扣除各种扣款

### 二、雇员分类

| 类型 | 支付方式 | 提交 |
|-----|---------|-----|
| 钟点工 | 按小时×报酬 | 时间卡 |
| 月薪员工 | 月薪 | 无 |
| 销售员工 | 月薪+销售提成 | 销售凭条 |

### 三、支付方式选择

- 银行卡直接存款
- 支票（邮寄或自取）
- 现金

### 四、协会成员

- 有些员工加入协会
- 每周扣除会费
- 可能产生服务费用

## 第十九章：薪水支付案例研究 - 实现

### 一、核心设计原则

> "刚学Java的时候，讲师提到写代码时要一个类一个方法只做一件事，遇到会变化的内容时要知道抽象出来。"

### 二、关键设计：支付策略抽象

```java
// 支付策略接口
public interface PaymentClassification {
    double calculatePay(Paycheck paycheck);
}

// 小时工支付
public class HourlyClassification implements PaymentClassification {
    private double hourlyRate;
    private List<TimeCard> timeCards;

    public double calculatePay(Paycheck paycheck) {
        // 计算每小时工的薪资
    }
}

// 月薪员工支付
public class SalariedClassification implements PaymentClassification {
    private double monthlyRate;
    public double calculatePay(Paycheck paycheck) {
        return monthlyRate;
    }
}

// 销售员工支付
public class CommissionedClassification implements PaymentClassification {
    private double monthlyRate;
    private double commissionRate;
    private List<SalesReceipt> salesReceipts;

    public double calculatePay(Paycheck paycheck) {
        // 计算销售提成
    }
}
```

### 三、雇员类设计

```java
public class Employee {
    private PaymentClassification classification;  // 支付策略
    private PaymentSchedule schedule;              // 支付日程
    private PaymentMethod method;                  // 支付方式
    private Affiliation affiliation;               // 协会关系
}
```

### 四、交易操作

```java
// 增加雇员
public abstract class AddEmployeeTransaction {
    public abstract Employee createEmployee();
}

// 更改雇员属性
public abstract class ChangeEmployeeTransaction {
    public abstract void update(Employee emp);
}
```

### 五、设计模式的应用

| 模式 | 应用 |
|-----|------|
| Strategy | PaymentClassification - 支付策略 |
| Command | Transaction - 事务操作 |
| Template Method | PayCalculation - 算法骨架 |
| Null Object | NoAffiliation - 无协会成员 |
| Factory | EmployeeFactory - 创建雇员 |

## 第十六至十九章核心总结

### 一、模式对比

| 模式 | 目的 | 机制 |
|-----|------|-----|
| Singleton | 控制实例数量 | 私有构造函数 |
| Monostate | 行为单一性 | 共享静态变量 |
| Null Object | 替代null | 空实现的接口 |

### 二、薪水支付系统设计核心

1. **支付策略抽象** - 将支付算法与员工分离
2. **交易命令化** - 所有操作建模为Transaction
3. **职责分离** - Classification/Schedule/Method分离

### 三、关键洞见

> "遇到会变化的内容时要知道抽象出来。"

> "Monostate模式不强制结构的单一性，而是强制行为的单一性。"

> "空对象模式可以完全替代真实对象，不会破坏程序行为。"
