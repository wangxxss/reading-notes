## 第二十章：包的设计原则

### 一、为什么要关注包（Package）

类是代码组织的基本单位，而包是类的组织方式。随着系统规模增大，我们需要关注：
- 哪些类应该放在一起？
- 包之间的依赖关系应该怎样？
- 如何让包结构支持可维护性和可扩展性？

---

### 二、包的內聚性原则（Cohesion Principles）

#### 1. 重用-发布等价原则（REP - Release-Reuse Equivalence Principle）

> "一个包的重用粒度必须和发布粒度相同。"

**含义**：
- 发布的最小单位是什么，重用的最小单位就是什么
- 不能只发布包中的某一个类，而要发布整个包
- 包的类应该一起被重用

**例子**：
- 如果发布了一个实用工具包，包含了10个类
- 用户可能只用了1个类，但必须引入整个包

#### 2. 共同重用原则（CRP - Common Reuse Principle）

> "一个包中的类应该一起被重用。如果重用了其中的一个类，就必须重用包中的所有类。"

**含义**：
- 包中的类是相互关联的
- 不要让用户只引入部分类
- 强迫用户引入整个包

**例子**：
- 数据访问包中的类通常一起使用
- 如果分离，会导致版本管理复杂

#### 3. 共同封闭原则（CNP - Common Closure Principle）

> "一个包中的类应该对同一类别的变化封闭。对一个包的改动，影响该包中的所有类；对其他包的改动，不应该影响该包。"

**含义**：
- 一起变化的类应该放在同一个包
- 分开变化的类应该放在不同包
- 减少变化的影响范围

---

### 三、包的耦合性原则（Coupling Principles）

#### 4. 无环依赖原则（ADP - Acyclic Dependencies Principle）

> "包的依赖关系图中不能出现环。"

**问题**：循环依赖

```
Package A → Package B → Package C → Package A
```

**解决方案**：
- 使用依赖倒置（DIP）
- 提取共同接口
- 重新设计包的边界

#### 5. 稳定依赖原则（SDP - Stable Dependencies Principle）

> "依赖的方向应该朝向稳定的方向。"

**不稳定度公式**：
```
I = Ca / (Ca + Ce)
```
- Ca：传入耦合（有多少包依赖我）
- Ce：传出耦合（我依赖了多少包）

**解读**：
- I=0：最稳定（被很多包依赖，但不依赖其他包）
- I=1：最不稳定（不被人依赖，但依赖很多包）

**原则**：依赖应该指向更稳定的包

#### 6. 稳定抽象等价原则（SAP - Stable Abstractions Principle）

> "一个包的抽象程度应该和它的稳定程度一致。"

**含义**：
- 稳定的包应该是抽象的（包含接口和抽象类）
- 不稳定的包应该是具体的（包含实现类）
- 稳定的包才值得依赖

---

### 四、包设计原则总结

| 原则 | 关注点 | 目标 |
|-----|-------|-----|
| REP | 重用粒度 | 保证发布的完整性 |
| CRP | 共同重用 | 减少引入不必要的类 |
| CNP | 共同封闭 | 隔离变化的影响 |
| ADP | 依赖关系 | 消除循环依赖 |
| SDP | 依赖方向 | 依赖稳定 |
| SAP | 抽象程度 | 稳定包要抽象 |

## 第二十一章：FACTORY模式

### 一、工厂模式的核心问题

**问题**：对象的创建逻辑不应该硬编码在使用对象的地方。

```java
// 问题：客户端直接依赖具体类
public class OrderProcessor {
    public void process() {
        Payment payment = new MySQLPayment();  // ❌ 依赖具体实现
    }
}
```

### 二、简单工厂（Simple Factory）

```java
// 工厂类
public class PaymentFactory {
    public static Payment create(String type) {
        if ("mysql".equals(type)) {
            return new MySQLPayment();
        } else if ("oracle".equals(type)) {
            return new OraclePayment();
        }
        throw new IllegalArgumentException();
    }
}

// 客户端
public class OrderProcessor {
    public void process() {
        Payment payment = PaymentFactory.create("mysql");
    }
}
```

### 三、工厂方法模式（Factory Method）

```java
// 工厂接口
public interface PaymentFactory {
    Payment createPayment();
}

// 具体工厂
public class MySQLPaymentFactory implements PaymentFactory {
    public Payment createPayment() {
        return new MySQLPayment();
    }
}

// 客户端使用抽象
public class OrderProcessor {
    private PaymentFactory factory;

    public OrderProcessor(PaymentFactory factory) {
        this.factory = factory;
    }

    public void process() {
        Payment payment = factory.createPayment();
    }
}
```

### 四、抽象工厂模式（Abstract Factory）

**场景**：创建一族相关对象

```java
// 抽象工厂
public interface DataAccessFactory {
    UserDAO createUserDAO();
    OrderDAO createOrderDAO();
    ProductDAO createProductDAO();
}

// MySQL工厂
public class MySQLFactory implements DataAccessFactory {
    public UserDAO createUserDAO() { return new MySQLUserDAO(); }
    public OrderDAO createOrderDAO() { return new MySQLOrderDAO(); }
    public ProductDAO createProductDAO() { return new MySQLProductDAO(); }
}

// Oracle工厂
public class OracleFactory implements DataAccessFactory {
    public UserDAO createUserDAO() { return new OracleUserDAO(); }
    public OrderDAO createOrderDAO() { return new OracleOrderDAO(); }
    public ProductDAO createProductDAO() { return new OracleProductDAO(); }
}
```

### 五、工厂模式的价值

| 好处 | 说明 |
|-----|------|
| **解耦创建逻辑** | 客户端不依赖具体类 |
| **易于替换实现** | 换工厂即换实现 |
| **易于测试** | 可以注入Mock工厂 |
| **符合DIP** | 依赖抽象而非具体 |

## 第二十二章：薪水支付案例研究（第2部分）

### 一、Factory模式应用

#### 创建雇员

```java
// 工厂接口
public interface EmployeeFactory {
    Employee createHourlyEmployee(String name, double rate);
    Employee createSalariedEmployee(String name, double salary);
    Employee createCommissionedEmployee(String name, double salary, double commission);
}

// 实现
public class PayrollFactory implements EmployeeFactory {
    public Employee createHourlyEmployee(String name, double rate) {
        Employee emp = new Employee(name);
        emp.setClassification(new HourlyClassification(rate));
        emp.setSchedule(new WeeklySchedule());
        emp.setMethod(new HoldMethod());
        return emp;
    }
    // ...
}
```

### 二、包的组织

#### 按职责划分包

```
payroll.domain          → Employee, PaymentClassification
payroll.transactions    → Transaction, AddEmployeeTransaction
payroll.factories       → EmployeeFactory
payroll.persistence     → EmployeeRepository
payroll.ui             → PayrollUI
```

#### 按稳定性划分

```
payroll.abstract       → 接口、抽象类（稳定）
payroll.concrete.hourly → 具体实现（不稳定）
```

### 三、设计原则检查

| 原则 | 应用 |
|-----|------|
| SRP | 每个类的职责单一 |
| OCP | 通过Factory扩展 |
| DIP | 依赖抽象EmployeeFactory |
| REP | 相关类在同一包 |
| ADP | 无循环依赖 |

## 第二十至二十二章核心总结

### 一、包设计原则速记

```
REP - 重用-发布等价：发布的最小单位=重用的最小单位
CRP - 共同重用：重用一个类=重用整个包
CNP - 共同封闭：一起变化的类放一起
ADP - 无环依赖：依赖图不能有环
SDP - 稳定依赖：依赖指向稳定的包
SAP - 稳定抽象等价：稳定的包要抽象
```

### 二、工厂模式对比

| 模式 | 适用场景 |
|-----|---------|
| 简单工厂 | 简单场景，一个工厂创建所有对象 |
| 工厂方法 | 需要多态，不同工厂创建不同产品 |
| 抽象工厂 | 创建一族相关对象 |

### 三、关键洞见

> "包的设计应该遵循內聚性原则，让相关的类在一起，不相关的类分开。"

> "工厂模式将对象创建与对象使用分离，客户端不依赖具体类。"

> "依赖应该指向稳定的方向，不稳定的包不应该被稳定的包依赖。"
