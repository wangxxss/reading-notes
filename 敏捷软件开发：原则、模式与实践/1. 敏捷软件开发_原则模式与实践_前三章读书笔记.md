# 《敏捷软件开发：原则、模式与实践》前三章读书笔记

## 第一章：敏捷实践

### 一、敏捷联盟与敏捷宣言

2001年，17位软件专家在犹他州雪鸟滑雪胜地共同签署了《敏捷宣言》，标志着敏捷软件开发的正式诞生。

**敏捷宣言四条核心价值观：**

| 左项（优先） | 右项（有价值） |
|------------|--------------|
| 人和交互重于 | 过程和工具 |
| 可以工作的软件重于 | 面面俱到的文档 |
| 客户合作重于 | 合同谈判 |
| 随时应对变化重于 | 遵循计划 |

> 原文："虽然右项也有其价值，但左项更加重要。"

### 二、四条价值观的深层含义

#### 1. 人和交互重于过程和工具
- 软件是人编出来的，离开了人是不行的
- 团队的作用十分重要
- 好的工具不能弥补糟糕的沟通

#### 2. 可以工作的软件重于面面俱到的文档
- 持续不断地频繁交付可以工作的软件比耗费巨大精力的文档更有价值
- 文档应该简洁、实用

#### 3. 客户合作重于合同谈判
- 成功的项目需要利益相关者（客户、业务人员）与开发人员的紧密合作
- 合同应该支持合作而非对抗

#### 4. 随时应对变化重于遵循计划
- 计划应该随着业务环境的变化而调整
- 僵化的计划会阻碍项目适应变化

### 三、敏捷十二条原则

1. 最优先的是通过尽早、持续交付有价值的软件满足客户
2. 欢迎需求变化，即使在开发后期
3. 经常交付可工作的软件，间隔越短越好
4. 业务人员和开发人员必须天天在一起工作
5. 围绕被激励的个人构建项目，提供环境和支持，信任他们
6. 面对面交谈是最高效的团队信息传递方式
7. **可工作的软件是进度的主要度量标准**
8. 敏捷过程倡导可持续的开发速度
9. 追求技术卓越和良好设计
10. **简洁：最大化工作量的艺术是减少工作量**
11. 自组织团队产生最好的架构和设计
12. 团队定期反思如何更高效，调整行为

### 四、核心洞见

> **"唯一能够加快进度的方法便是缩减范围。不要经受不住诱惑盲目冲刺。"**

> **敏捷开发人员知道要做什么，是因为：**
> 1. 遵循敏捷实践去发现问题
> 2. 应用设计原则去诊断问题
> 3. 应用适当的设计模式去解决问题
>
> **软件开发的这三个方面的相互作用就是设计。**

## 第二章：极限编程概述

### 一、极限编程（XP）简介

极限编程由 Kent Beck 在1996年提出，是最早成形的敏捷方法论之一。XP由一系列简单却互相依赖的实践组成，这些实践结合在一起形成了**胜于部分结合的整体**。

### 二、XP的四大价值观

1. **沟通（Communication）** - 团队成员、项目与客户间的紧密沟通
2. **简单性（Simplicity）** - 只做当前需要的事，不过度设计
3. **反馈（Feedback）** - 持续获取反馈，快速调整
4. **勇气（Courage）** - 勇于重构、勇于接受变化

### 三、XP的十二项核心实践

#### 2.1.1 完整团队（Whole Team）
- 团队应包含所有角色：开发、测试、业务人员
- 客户作为团队成员全职或接近全职参与

#### 2.1.2 用户故事（User Stories）
- 用户故事是客户表达的**简短需求描述**
- 用卡片或电子形式记录
- 由客户和开发人员在计划会议上共同编写
- 格式："作为一个<角色>，我想要<功能>，以便<价值>"

#### 2.1.3 短交付周期（Short Cycles）
**发布计划（Release Planning）：**
- XP项目通常每两周交付一次可工作的软件
- 每次迭代结束时，向利益相关者演示系统，获取反馈

**迭代计划（Iteration Planning）：**
- 每次迭代通常耗时两周
- 由一组用户故事组成
- 客户选择要实现的素材，开发人员估算工作量

#### 2.1.4 验收测试（Acceptance Tests）
- 验收测试由客户编写，验证系统功能
- 使用自动化测试框架
- 验收测试应该尽可能早地开始编写

#### 2.1.5 结对编程（Pair Programming）
- 所有产品软件由两个程序员并排坐在同一台机器上构建
- 一人写代码，一人review/思考
- 定期交换角色
- **好处**：知识共享、质量提升、代码审查

#### 2.1.6 测试驱动开发（TDD - Test-Driven Development）
**做法：**
1. 在设计程序前先设计测试方案
2. 不多实现一个功能，除非少了该功能会导致测试失败
3. 不多实现一行代码，除非少了这行代码会导致测试失败

**"红-绿-重构"循环：**
- 红：编写一个失败的测试
- 绿：编写最小代码使测试通过
- 重构：优化代码结构，保持测试通过

**作用：**
- 每一项功能都有测试验证正确性
- 迫使我们从不同视角观察程序
- 迫使我们对程序解耦

#### 2.1.7 集体所有权（Collective Ownership）
- 任何成员都可以在任何时候改进任何代码
- 没有"这是别人的代码"的说法
- 鼓励知识在团队中传播

#### 2.1.8 持续集成（Continuous Integration）
- 团队总是使系统完整地被集成
- 有人check in后，其他人要立即集成
- 保持代码库的持续可发布状态

#### 2.1.9 可持续的开发速度（Sustainable Pace）
- 反对加班文化
- 长时间高强度工作会降低效率和质量
- 团队应该能够长期保持稳定的工作节奏

#### 2.1.10 开放的工作空间（Open Workspace）
- 团队成员坐得近，便于沟通
- 配备大桌子、白板等便于讨论的设施

#### 2.1.11 计划游戏（Planning Game）
- 将业务决策和技术决策分离
- 客户决定需求优先级和交付日期
- 开发人员估算成本和技术方案

#### 2.1.12 重构（Refactoring）
- **定义**：在不改变代码行为的前提下，对其进行一系列小的改造
- 每个改造微不足道，但累积起来能显著改进系统设计和架构
- 每次重构后必须运行单元测试，保证行为不变

### 四、XP三条指导原则（Mantras）

1. **考虑能工作的最简单的事情** - Don't repeat yourself
2. **你不需要它** - YAGNI (You Aren't Gonna Need It)
3. **让代码充满激情** - 在简单设计与功能实现间找到平衡

## 第三章：计划

### 一、敏捷计划的核心思想

> "当你能够度量你所说的，并且能够用数字去表达它时，就表示你了解它了；若你不能度量它，不能用数字去表达它，那么说明你的知识是匮乏的、不能令人满意的。" —— 开尔文勋爵，1883

### 二、初始探索（Initial Exploration）

1. **时机**：项目开始时，开发人员和客户商讨新系统的情况
2. **目标**：确定所有真正重要的用户素材
3. **态度**：不试图确定所有用户素材，因为需求会随项目进展变化
4. **持续性**：特性发现过程会持续到项目完成
5. **形式**：把特性分解成用户故事，写在索引卡片之类的东西上

### 三、发布计划（Release Planning）

1. **参与者**：开发人员和客户
2. **任务**：共同估算用户素材，排列优先级
3. **原则**：
   - 开发人员共同估算，估算是相对的，不是绝对的
   - 客户选择他们想要在某个发布中实现的素材
   - 客户大致确定素材的实现顺序
4. **发布条件**：当利益相关者满意时发布

### 四、迭代计划（Iteration Planning）

**迭代时长**：通常为两周

**关键规则：**
1. 开发人员和客户决定迭代顺序
2. 迭代期间的实现顺序属于技术范畴，**由开发人员决定**（并行还是串行）
3. **一旦迭代开始，客户不能再改变迭代范围内的素材**
4. 即使没有完成所有素材，迭代也要在指定日期结束

**速度因子：**
- 根据已完成的素材估算值计算
- 这个速度会用于指导下一次迭代的规划

### 五、任务计划（Task Planning）

1. **拆分**：对素材进行拆解，变成一个个小任务
2. **粒度**：每个任务大约4-16小时工作量
3. **参与**：开发人员和客户一起详细拆解，尽可能列出所有任务
4. **形式**：在白板、活动挂图等媒介上列出任务
5. **认领**：开发人员自行选择想要完成的任务
6. **估算**：使用随意的任务点数对任务进行估算

### 六、"完成"的定义（Definition of "Done"）

- 明确"完成"的含义非常重要
- 建议：每个任务完成后都要集成到系统中并通过所有测试

### 七、迭代中期检查

- 迭代进行到一半时，半数素材应该被完成
- 如果没有完成，团队会设法重新分配任务和职责
- 这是对速度和能力的持续校准

### 八、核心要点总结

| 阶段 | 参与者 | 关键产出 |
|-----|-------|---------|
| 初始探索 | 开发人员+客户 | 用户故事列表 |
| 发布计划 | 开发人员+客户 | 发布顺序和估算 |
| 迭代计划 | 开发人员+客户 | 本次迭代的素材和顺序 |
| 任务计划 | 开发人员 | 任务列表和人员分配 |

## 三章核心总结

### 敏捷开发的核心要素

```
┌─────────────────────────────────────────────────────────┐
│                      敏捷开发                             │
├─────────────────────────────────────────────────────────┤
│  价值观 ──────► 原则 ──────► 实践                         │
│  (宣言)        (12条)     (XP的具体做法)                  │
└─────────────────────────────────────────────────────────┘
```

### 最重要的洞见

1. **人是最重要的**：过程和工具是死的，人是活的
2. **软件要能工作**：文档再多，不能运行的软件没有价值
3. **客户要合作**：合同不能替代真正的参与
4. **拥抱变化**：变化是机会，不是障碍
5. **保持可持续的速度**：长期稳定胜过短期冲刺
6. **设计原则指导实践**：敏捷实践需要设计原则的支撑
