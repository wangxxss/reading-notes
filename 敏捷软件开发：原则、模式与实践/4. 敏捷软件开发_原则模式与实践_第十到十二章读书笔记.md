# 《敏捷软件开发：原则、模式与实践》第十至十二章读书笔记

---

## 第十章：Liskov替换原则（LSP）

### 一、LSP定义

> **"子类型（subtype）必须能够替换掉它们的基类型（base type）。"**

这是由计算机科学家 **芭芭拉·利斯科夫（Barbara Liskov）** 于1987年提出的原则。

Barbara Liskov 是美国计算机科学家，2008年图灵奖获得者、2004年冯诺依曼奖得主。

---

### 二、LSP的核心含义

#### 字面理解
如果你写了一段代码，是基于一个父类/接口来操作的。那么当你把这段代码中的父类对象，替换成它的任何一个子类对象时，**程序的行为都不应该出问题**（比如崩溃、报错、产生意外结果）。

#### IS-A关系的真正含义

> "换句话说，子类必须完全遵守父类定下的'契约'或'承诺'。"

**传统观点**：
- "正方形是一个矩形" → 正方形类应该从矩形类派生
- 这是基于**属性**的IS-A关系

**LSP观点**：
- IS-A关系应该是基于**行为方式**的
- 子类的行为必须符合基类对使用者承诺的行为
- **"正方形是一个矩形"这个说法是错误的**，因为行为不一致

---

### 三、正方形和矩形：违反LSP的经典案例

#### 问题分析

```java
// 矩形的定义
class Rectangle {
    void setWidth(double w) { width = w; }
    void setHeight(double h) { height = h; }
    double getArea() { return width * height; }
}

// 正方形从矩形派生（看似合理）
class Square extends Rectangle {
    @Override
    void setWidth(double w) {
        width = w;
        height = w;  // 强制两边相等
    }
    @Override
    void setHeight(double h) {
        width = h;
        height = h;  // 强制两边相等
    }
}
```

#### 违反LSP的场景

```java
void g(Rectangle r) {
    r.setWidth(5);   // 设置宽度为5
    r.setHeight(4);  // 设置高度为4
    // 期望面积 = 5 * 4 = 20
    // 但如果是Square：
    // setWidth(5)后 width=5, height=5
    // setHeight(4)后 width=4, height=4
    // 实际面积 = 4 * 4 = 16 ❌
}
```

**问题**：函数 `g` 假设 Rectangle 的 width 和 height 是独立的，但 Square 违反了这个假设。

#### 结论

> "对于g来说，Square不能替换Rectangle，因此Square和Rectangle之间的关系是违反LSP的。"

---

### 四、基于契约设计（Design By Contract, DBC）

#### 什么是契约

契约通过为每个函数声明**前置条件**和**后置条件**来指定：

| 条件类型 | 说明 | 谁确保 |
|---------|------|-------|
| 前置条件 | 调用者必须满足的约束 | 调用者 |
| 后置条件 | 方法必须保证的结果 | 实现者 |

#### LSP与契约

> "LSP清楚地指出，OOD中IS-A关系是就**行为方式**而言的，行为方式是可以进行合理假设的，是客户程序所依赖的。"

**子类的后置条件不能比父类更宽松**：
- 父类承诺返回 A
- 子类不能只返回 A 的子集

---

### 五、违反LSP的信号

#### 1. 退化函数（Degenerate Functions）
派生类中有退化函数，从基类"去除"了某些功能。

#### 2. 抛出异常
派生类中的方法会抛出基类不允许的异常。

#### 3. 违反隐式约束
子类的方法虽然技术上符合签名，但违反了基类对行为的隐式假设。

---

### 六、违反LSP的实际例子

#### 鸵鸟不是鸟

```java
class Bird {
    void setFlySpeed(double speed) { /* 设置飞行速度 */ }
    double fly(double distance) { return distance / speed; }
}

class Ostrich extends Bird {
    @Override
    void setFlySpeed(double speed) {
        // 鸵鸟不会飞，强制设置为0
        super.setFlySpeed(0);
    }
}
```

**问题**：`Bird` 的客户假设 `setFlySpeed` 总是能设置有意义的值，但 `Ostrich` 违反了这个假设。

---

### 七、违反LSP的解决方案

#### 方案：提取公共部分作为抽象

把两个类的公共部分提取出来作为一个**抽象基类**：

```
原方案（违反LSP）：
        Rectangle
           △
           │
         Square

改进方案：
        Shape (抽象)
           △
           │
    ┌──────┴──────┐
    │             │
Rectangle      Square
```

---

### 八、LSP与OCP的关系

> "LSP是使OCP成为可能的主要原则之一。正是子类型的**可替换性**才使得使用基类类型的模块在无需修改的情况下就可以扩展。"

**没有LSP，OCP就无法实现**：
- OCP要求对扩展开放
- 扩展意味着新的子类可以被使用
- 但如果子类不能正确替换基类，就会破坏OCP

## 第十一章：依赖倒置原则（DIP）

### 一、DIP定义

> **"依赖倒置原则"包含两层含义：**
> 1. **高层模块不应该依赖于低层模块，两者都应该依赖于抽象**
> 2. **抽象不应该依赖于细节，细节应该依赖于抽象**

---

### 二、传统软件结构的问题

#### 传统的分层方式

```
┌─────────────────┐
│  高层策略模块    │  ← 依赖 →
└─────────────────┘
┌─────────────────┐
│  低层具体实现    │
└─────────────────┘
```

**问题**：
- 高层模块依赖于低层模块
- 高层模块被低层模块的细节所绑定
- 低层模块的变化直接影响高层模块

#### 传统观点的错误假设

1. 策略（Policy）应该依赖于细节（Details）
2. 现实世界中，策略和细节是分离的
3. 但传统设计让策略依赖于细节

---

### 三、DIP的核心理念

> **"面向抽象编程，而非面向实现编程。"**

#### DIP的软件结构

```
┌─────────────────┐
│  高层策略模块    │
└────────┬────────┘
         │ 依赖抽象
         ▼
┌─────────────────┐
│  抽象接口/抽象类  │  ← 稳定的
└────────┬────────┘
         │ 依赖抽象
         ▼
┌─────────────────┐
│  低层具体实现    │
└─────────────────┘
```

**关键变化**：
- 依赖关系**倒置**了
- 低层现在依赖于高层定义的抽象

---

### 四、DIP的具体指导

#### DIP规则

1. **任何变量都不应该持有指向具体类的引用**
   - 改用抽象基类或接口

2. **任何类都不应该从具体类派生**
   - 从抽象类或接口派生

3. **任何方法都不应该覆写基类中已经实现的方法**
   - 继承树顶端应该是抽象

4. **所有依赖关系都应该终止于抽象类或接口**

---

### 五、DIP代码示例

#### 违反DIP的设计

```java
// 高级策略
class OrderProcessor {
    private MySQLDatabase database;  // ❌ 依赖具体类

    void process(Order order) {
        database.save(order);
    }
}

// 具体低层实现
class MySQLDatabase {
    void save(Order order) { /* MySQL实现 */ }
}
```

#### 符合DIP的设计

```java
// 抽象接口（高层定义）
interface Repository {
    void save(Order order);
}

// 低层实现
class MySQLDatabase implements Repository {
    public void save(Order order) { /* MySQL实现 */ }
}

class MongoDBDatabase implements Repository {
    public void save(Order order) { /* MongoDB实现 */ }
}

// 高级策略
class OrderProcessor {
    private Repository repository;  // ✅ 依赖抽象

    OrderProcessor(Repository repository) {
        this.repository = repository;
    }

    void process(Order order) {
        repository.save(order);
    }
}
```

---

### 六、DIP的层次化

#### 什么是层次化

```
┌────────────────────────────────┐
│     应用程序专用类/策略         │  第5层
├────────────────────────────────┤
│     业务逻辑/核心框架接口        │  第4层
├────────────────────────────────┤
│        抽象接口实现             │  第3层
├────────────────────────────────┤
│        具体工具类               │  第2层
├────────────────────────────────┤
│          低层工具              │  第1层
└────────────────────────────────┘
```

**DIP要求**：第5层不应依赖第1层，应该穿过中间层依赖第4层。

---

### 七、DIP与好莱坞原则

> **"不要给我们打电话，我们会给你打电话。"**

好莱坞原则（Hollywood Principle）与DIP有相似的哲学：
- 高层组件调用低层组件是被禁止的
- 低层组件"注册"到高层，高层决定何时调用

---

### 八、DIP的实际应用：插件架构

```
┌─────────────────────────────────┐
│      应用程序核心（高层）        │
│                                 │
│  ┌───────────────────────────┐  │
│  │    插件接口（稳定的抽象）    │  │
│  └───────────────────────────┘  │
└───────────────┬─────────────────┘
                │
    ┌───────────┼───────────┐
    │           │           │
┌───┴───┐   ┌───┴───┐   ┌───┴───┐
│ 插件A  │   │ 插件B  │   │ 插件C  │
└───────┘   └───────┘   └───────┘
```

应用程序不依赖任何具体插件，只依赖插件接口。

---

### 九、DIP的价值

| 好处 | 说明 |
|-----|------|
| **灵活性** | 低层实现可以替换 |
| **可测试性** | 可以注入Mock对象 |
| **可扩展性** | 添加新实现无需修改高层 |
| **解耦** | 减少模块间耦合 |

## 第十二章：接口隔离原则（ISP）

### 一、ISP定义

> **"客户端不应该被强迫去依赖它们并不需要的接口。"**

原文：
> "CLIENTS SHOULD NOT BE FORCED TO DEPEND UPON INTERFACES THAT THEY DO NOT USE."

---

### 二、什么是接口污染

#### "胖"接口的问题

如果一个类的接口不是**内聚的（cohesive）**，就说这个类具有"胖接口"。

**接口污染示例**：

```java
// 一个"胖"门接口
interface Door {
    void lock();
    void unlock();
    boolean isOpen();

    // ❌ 定时器相关的方法不应该在这里
    void registerTimerClient(TimerClient client);
    void unregisterTimerClient(TimerClient client);
}
```

**问题**：
- `Door` 接口被 `TimerClient` 相关方法污染了
- 只有 `TimedDoor` 需要定时功能
- 普通 `Door` 的实现必须实现这些不需要的方法

---

### 三、分离客户就是分离接口

#### ISP的核心洞察

> "客户程序是分离的，所以接口也应该保持分离。"

**原因**：
- 不同客户程序使用不同的方法
- 当一个客户程序要求改变接口时，会影响其他客户程序
- 应该根据客户程序的需求来分离接口

---

### 四、ISP的解决方案

#### 方案：拆分大接口为多个小接口

```
违反ISP的方案：
        Door (胖接口)
     ┌────┼────┐
     │    │    │
TimedDoor  ... ...

符合ISP的方案：

   Door          TimerClient (分离的)
     △               △
     │               │
     │    ┌──────────┘
     │    │
TimedDoor
```

---

### 五、ISP代码示例：Door + Timer

#### 违反ISP：Door继承TimerClient

```java
// Timer的接口
interface TimerClient {
    void Timeout();
}

// Door被污染
interface Door extends TimerClient {
    void lock();
    void unlock();
    boolean isOpen();
}

// 普通门实现
class SimpleDoor implements Door {
    @Override
    void lock() { }
    @Override
    void unlock() { }
    @Override
    boolean isOpen() { }

    // ❌ 不需要定时功能，但必须实现
    @Override
    void Timeout() { }  // 退化实现
}
```

#### 符合ISP：分离接口

```java
// 独立的Door接口
interface Door {
    void lock();
    void unlock();
    boolean isOpen();
}

// 独立的TimerClient接口
interface TimerClient {
    void Timeout();
}

// TimedDoor实现两个独立接口
class TimedDoor implements Door, TimerClient {
    @Override
    public void lock() { }
    @Override
    public void unlock() { }
    @Override
    public boolean isOpen() { }

    @Override
    public void Timeout() {
        // 定时器超时的真正实现
        this.lock();
    }
}

// 普通门保持简单
class SimpleDoor implements Door {
    @Override
    public void lock() { }
    @Override
    public void unlock() { }
    @Override
    public boolean isOpen() { }
}
```

### 六、ISP与ATM示例

#### 问题背景

ATM需要处理多种事务：
- 存款（Deposit）
- 取款（Withdrawal）
- 转账（Transfer）

每种事务都需要不同的UI操作。

#### 违反ISP的设计

```
           Transaction (抽象基类)
           ┌────┬────┐
           │         │
Deposit   Withdrawal  Transfer
   │
   ├── RequestDepositAmount()   ← 其他事务不用
   ├── ShowDepositReceipt()     ← 其他事务不用
```

#### 符合ISP的设计

```
       Transaction                UI接口分离
           │                    ┌──────────┐
           │                    │DepositUI │
           │                    └──────────┘
    ┌──────┼──────┐              ┌──────────┐
    │      │      │              │WithdrawalUI│
Deposit Withdrawal Transfer     └──────────┘
                                  ┌──────────┐
                                  │TransferUI│
                                  └──────────┘
```

**好处**：
- 每种事务只依赖它需要的UI接口
- UI变化不会影响不相关的Transaction

### 七、类接口与对象接口

#### 访问对象的方式

对象的客户程序不必通过对象的接口来访问它，可以通过：
1. **委托（ Delegation）** - 使用适配器
2. **多重继承（Multiple Inheritance）**
3. **基类指针** - 直接访问对象的基类部分

---

### 八、ISP与其他SOLID原则的关系

| 关系 | 说明 |
|-----|------|
| ISP 与 SRP | ISP是接口级别的SRP；SRP关注类的职责，ISP关注接口的职责 |
| ISP 与 DIP | DIP是关于依赖方向的，ISP是关于接口大小的 |
| ISP 与 OCP | 胖接口会阻碍扩展，ISP通过分离接口支持OCP |

---

### 九、ISP的实践建议

#### 何时考虑ISP

1. **接口包含不相关的方法** - 接口承担了多个不相关职责
2. **实现类实现不需要的方法** - 被迫提供退化实现
3. **接口变更频繁** - 一个客户的变化影响所有客户

#### ISP的平衡

> "ISP承认存在有一些对象，它们确实不需要内聚的接口。但是ISP建议客户程序不应该看到它们作为单一的类存在。"

**建议**：
- 内部可以是大而全的类
- 但对外暴露给不同客户的应该是**专用的小接口**

## 第十至十二章核心总结

### 一、三大原则对比

| 原则 | 关注点 | 问题 | 解决方案 |
|-----|-------|------|---------|
| **LSP** | 子类型替换 | 子类不能正确替换基类 | 基于行为设计，遵守契约 |
| **DIP** | 依赖方向 | 高层依赖低层 | 依赖抽象，倒置方向 |
| **ISP** | 接口大小 | 胖接口污染 | 分离接口，客户专用 |

---

### 二、LSP核心要点

```
LSP核心：子类必须能替换基类

正方形不是长方形 ❌
（因为行为不一致）

正方形是一个能计算面积的形状 ✅
（基于行为的一致性）
```

---

### 三、DIP核心要点

```
传统：高层 → 低层（依赖具体）
DIP：  高层 → 抽象 ← 低层（依赖倒置）

关键规则：
• 变量不持有具体类引用
• 类不从具体类派生
• 方法不覆写已实现的方法
```

---

### 四、ISP核心要点

```
ISP核心：客户端不应该依赖不使用的方法

胖接口 → 拆分为多个专用接口

对象内部可以复杂
对外接口必须内聚
```

### 六、关键洞见

> **"术语'IS-A'的含义过于宽泛以至于不能作为子类型的定义。子类型的正确定义是'可替换性的'。"**

> **"由于遵循OCP要花很多时间精力去创建正确的抽象，所以在有限的成本下，把OCP尽可能应用在可能发生的变化，并且直到发生变化时才采取行动。"**

> **"ISP承认存在有一些对象，它们确实不需要内聚的接口。但是ISP建议客户程序不应该看到它们作为单一的类存在。"**

### 七、设计检查清单

| 问题 | 检查项 |
|-----|-------|
| LSP | 子类可以替换基类吗？子类行为是否一致？ |
| DIP | 依赖是否终止于抽象？是否依赖具体类？ |
| ISP | 接口是否内聚？客户端是否被迫实现不需要的方法？ |

## 附录：SOLID五大原则速查

```
S - SRP (Single Responsibility Principle)
  单一职责原则
  一个类只有一个变化的原因

O - OCP (Open-Closed Principle)
  开放封闭原则
  对扩展开放，对修改封闭

L - LSP (Liskov Substitution Principle)
  里氏替换原则
  子类必须能替换基类

I - ISP (Interface Segregation Principle)
  接口隔离原则
  客户端不应依赖不使用的方法

D - DIP (Dependency Inversion Principle)
  依赖倒置原则
  依赖抽象而非具体
```
