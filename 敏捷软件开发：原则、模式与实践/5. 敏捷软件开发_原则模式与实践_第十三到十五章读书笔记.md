## 第十三章：COMMAND模式和ACTIVE OBJECT模式

### 一、COMMAND模式

#### 模式定义

> "通过对命令（Command）概念的封装，解除了系统与逻辑、互联关系、设备之间的耦合。"

#### 核心结构

COMMAND模式的核心非常简单：

```java
public interface Command {
    void execute();
}
```

这就是COMMAND模式的全部！

#### 模式结构

```
┌─────────────────┐
│     Client      │  客户（创建命令）
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│    Command       │  命令接口
│  (execute())    │
└────────┬────────┘
         │
         │ implements
         ▼
┌─────────────────┐
│ ConcreteCommand │  具体命令
│  execute() {    │
│    receiver.    │
│    action();    │
│  }             │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│    Receiver     │  接收者（执行实际操作）
└─────────────────┘
```

#### 生活类比

**场景**：经理张三叫leader王二去开发项目，王二安排李四去开发功能A。李四何时执行、怎么执行就是他自己的事情。

- Client（经理）：创建命令
- Invoker（leader）：触发命令
- Command（命令接口）：封装请求
- Receiver（李四）：实际执行者

#### 薪水支付系统案例

在薪水支付系统中，Transaction（事务）被建模为Command：

```java
public interface Transaction {
    void execute();
}

public class AddEmployeeTransaction implements Transaction {
    private Employee emp;
    public void execute() {
        // 添加雇员的实际逻辑
    }
}
```

**好处**：
- 事务可以被排队、延迟执行
- 事务可以撤销（undo）
- 事务可以组合

### 二、ACTIVE OBJECT模式

#### 什么是ACTIVE OBJECT

ACTIVE OBJECT模式是一种**并发设计模式**，它解耦了方法的**执行**和方法的**调用**。

**核心思想**：
- 方法调用在调用者线程执行
- 方法执行在Active Object自己的线程执行
- 通过命令队列调度

#### 为什么需要ACTIVE OBJECT

传统方式的问题：
```java
// 阻塞调用，调用者必须等待完成
device.send(data);  // 可能很慢
// 继续执行其他代码  // 必须等待send完成
```

ACTIVE OBJECT的解决：
```java
// 非阻塞调用，立即返回
device.sendAsync(data);  // 立即返回
// 继续执行其他代码  // 不需要等待
```

#### 模式结构

```
┌──────────────────────────────────────────────────┐
│              ActiveObjectEngine                  │
│  ┌──────────────────────────────────────────┐   │
│  │  List<Command> commands  (命令队列)       │   │
│  └──────────────────────────────────────────┘   │
│                       │                         │
│                       ▼                         │
│              run() 循环处理命令                  │
└──────────────────────────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│                Command接口                        │
│              void execute()                       │
└──────────────────────────────────────────────────┘
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ SleepCommand│ │DelayedTyper│ │  ...其他命令 │
└─────────────┘ └─────────────┘ └─────────────┘
```

#### SleepCommand示例

```java
public class SleepCommand implements Command {
    private long sleepTime;
    private ActiveObjectEngine engine;
    private Command wakeupCommand;
    private Date startTime;
    private boolean started = false;

    public void execute() {
        Date currentTime = new Date();
        if (!started) {
            started = true;
            startTime = currentTime;
            engine.addCommand(this);  // 把自己重新放回队列
        } else {
            long elapsed = currentTime.getTime() - startTime.getTime();
            if (elapsed < sleepTime) {
                engine.addCommand(this);  // 时间还没到，再等
            } else {
                engine.addCommand(wakeupCommand);  // 时间到了，执行唤醒命令
            }
        }
    }
}
```

#### 实际应用场景

| 场景 | 说明 |
|-----|------|
| GUI事件处理 | 按钮点击事件放入队列异步处理 |
| 打印机队列 | 打印任务排队依次执行 |
| 数据库请求 | SQL命令排队执行 |
| 定时器 | 延时操作调度 |

### 三、命令模式的价值

#### COMMAND模式的好处

1. **解耦发送者和接收者**
   - 发送者不需要知道接收者是谁
   - 命令可以被延迟执行

2. **支持撤销/重做**
   - 命令可以存储状态来撤销操作

3. **支持日志记录**
   - 命令可以记录操作历史

4. **支持事务**
   - 命令可以组合成事务

#### COMMAND vs PROCEDURE CALL

| 对比 | 过程调用 | 命令模式 |
|-----|---------|---------|
| 时机 | 同步，调用者等待 | 可以异步执行 |
| 耦合 | 调用者绑定接收者 | 完全解耦 |
| 灵活性 | 低 | 高 |
| 延迟执行 | 不支持 | 支持 |

## 第十四章：TEMPLATE METHOD模式和STRATEGY模式

### 一、两种模式的对比

#### 核心区别

| 模式 | 解决方案 | 机制 |
|-----|---------|------|
| **Template Method** | 继承 | 子类覆盖父类的方法 |
| **Strategy** | 委托 | 对象组合，将算法委托给其他对象 |

#### Template Method模式

**定义**：
> "定义一个操作中的算法骨架，将某些步骤延迟到子类中。"

**核心思想**：**继承**

```java
// 抽象父类 - 定义算法骨架
abstract class ApplicationFramework {
    // 模板方法 - 算法骨架
    public void run() {
        initialize();
        while (!done()) {
            idle();
        }
        cleanup();
    }

    // 这些方法由子类实现
    protected abstract void initialize();
    protected abstract boolean done();
    protected abstract void idle();
    protected abstract void cleanup();
}

// 具体子类
class MyApp extends ApplicationFramework {
    protected void initialize() { /* 实现 */ }
    protected boolean done() { return false; }
    protected void idle() { /* 实现 */ }
    protected void cleanup() { /* 实现 */ }
}
```

#### Strategy模式

**定义**：
> "定义一系列算法，把它们一个个封装起来，并且使它们可以相互替换。"

**核心思想**：**委托**

```java
// 算法接口
interface Strategy {
    void solve();
}

// 具体算法实现
class QuickSort implements Strategy {
    public void solve() { /* 快速排序 */ }
}

class BubbleSort implements Strategy {
    public void solve() { /* 冒泡排序 */ }
}

// 上下文 - 使用策略
class Sorter {
    private Strategy strategy;

    public Sorter(Strategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(Strategy strategy) {
        this.strategy = strategy;
    }

    public void execute() {
        strategy.solve();
    }
}
```

### 二、温度打印案例

#### Template Method实现

```java
abstract class TemperatureReader {
    // 模板方法
    public void process() {
        double[] temps = readTemperatures();
        double avg = calculateAverage(temps);
        printReport(temps, avg);
    }

    abstract double[] readTemperatures();
    abstract void printReport(double[] temps, double avg);

    double calculateAverage(double[] temps) {
        // 通用的平均计算
        double sum = 0;
        for (double t : temps) sum += t;
        return sum / temps.length;
    }
}
```

#### Strategy实现

```java
interface TemperatureReader {
    double[] read();
}

interface TemperatureReporter {
    void report(double[] temps, double avg);
}

// 使用策略
class TemperatureProcessor {
    private TemperatureReader reader;
    private TemperatureReporter reporter;

    public TemperatureProcessor(Strategy r, Strategy p) {
        reader = (TemperatureReader) r;
        reporter = (TemperatureReporter) p;
    }
}
```

### 三、冒泡排序案例

#### 问题

需要对不同类型的数据进行排序，但排序算法可能变化。

#### Template Method方案

```java
abstract class BubbleSorter {
    protected int length;
    protected Object[] array;

    public void sort(Object[] arr) {
        this.array = arr;
        this.length = arr.length;
        for (int i = length - 1; i >= 0; i--) {
            for (int j = 0; j < i; j++) {
                if (outOfOrder(j, j + 1)) {
                    swap(j, j + 1);
                }
            }
        }
    }

    protected abstract boolean outOfOrder(int i, int j);
    protected abstract void swap(int i, int j);
}

class IntBubbleSorter extends BubbleSorter {
    protected boolean outOfOrder(int i, int j) {
        return ((Integer[]) array)[i] > ((Integer[]) array)[j];
    }

    protected void swap(int i, int j) {
        Integer temp = ((Integer[]) array)[i];
        ((Integer[]) array)[i] = ((Integer[]) array)[j];
        ((Integer[]) array)[j] = temp;
    }
}
```

#### Strategy方案

```java
interface SortStrategy {
    void sort(Object[] arr);
}

class IntSortStrategy implements SortStrategy {
    public void sort(Object[] arr) {
        // 具体的排序算法
    }
}

class Sorter {
    private SortStrategy strategy;

    public Sorter(SortStrategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(SortStrategy strategy) {
        this.strategy = strategy;
    }

    public void executeSort(Object[] arr) {
        strategy.sort(arr);
    }
}
```

### 四、两种模式的取舍

#### 选择Template Method的场景

1. 需要**继承**来共享通用行为
2. 算法骨架相对**稳定**
3. 子类差异在于**步骤实现**
4. **性能**要求高（无委托开销）

#### 选择Strategy的场景

1. 需要**动态切换**算法
2. 多种算法**平等**，可替换
3. 算法需要**封装**（隐藏实现）
4. 遵循**DIP原则**（依赖抽象）

#### 关键区别

> "Template Method模式使用继承来解决问题，而Strategy模式使用的则是委托。"

> "Strategy模式比Template Method模式多提供了一个额外的好处：尽管Template Method模式允许一个通用算法操纵多个可能的具体实现，但是由于Strategy模式完全遵循DIP原则，从而允许每个具体实现都可以被多个不同的通用算法操纵。"

---

### 五、模式不是万能药

#### 过度使用模式的问题

> "没有遇到变化的时候，就用模式，当然会变得臃肿了。"

**警告信号**：
- 只有一种实现，还在用模式
- 变化还没有发生
- 增加了不必要的复杂性

**原则**：
> "YAGNI - You Aren't Gonna Need It"（你不需要它）

## 第十五章：FACADE模式和MEDIATOR模式

### 一、FACADE模式（门面模式）

#### 模式定义

> "为子系统中的一组接口提供一个一致的界面，Facade定义了一个高层接口，使得子系统更易于使用。"

#### 生活类比：医院

**没有Facade**：
- 病人必须与每个部门打交道
- 先挂号 → 门诊 → 划价 → 缴费 → 检查 → 取药
- 极其繁琐

**有Facade（接待员）**：
- 病人只需要找接待员
- 接待员负责协调所有部门
- 病人省心

```
无Facade:                     有Facade:
病人 → 挂号 → 门诊 → ...       病人 → 接待员(Facade)
                               接待员 → 各部门
```

#### 模式结构

```
┌─────────────────────────┐
│        Client           │
└────────────┬────────────┘
             │ 调用门面
             ▼
┌─────────────────────────┐
│       Facade            │  门面角色
│  operation() {         │  委托给子系统
│    subsystemA.op1();    │
│    subsystemB.op2();    │
│  }                      │
└────────────┬────────────┘
             │
    ┌────────┴────────┐
    ▼                 ▼
┌─────────┐      ┌─────────┐
│Subsystem│      │Subsystem│
│   A     │      │   B     │
└─────────┘      └─────────┘
```

#### 代码示例：订单系统

**无Facade**：
```java
// 客户端直接调用所有子系统
validator.validate(order);
inventory.checkStock(order);
payment.processPayment(order);
notify.sendConfirmEmail(order);
shipping.scheduleShipment(order);
```

**有Facade**：
```java
// 客户端只需要调用门面
public class OrderFacade {
    private OrderValidator validator;
    private InventoryService inventory;
    private PaymentService payment;
    private NotificationService notify;
    private ShippingService shipping;

    public void placeOrder(Order order) {
        if (!validator.validate(order)) return;
        inventory.checkStock(order);
        payment.processPayment(order);
        notify.sendConfirmEmail(order);
        shipping.scheduleShipment(order);
    }
}

// 客户端简洁调用
OrderFacade facade = new OrderFacade();
facade.placeOrder(order);
```

---

### 二、FACADE模式的优点

#### 1. 减少系统相互依赖

```
无Facade：                    有Facade：
外界 → 子系统A                外界 → Facade → 子系统A
外界 → 子系统B                外界 → Facade → 子系统B
外界 → 子系统C                外界 → Facade → 子系统C
（强耦合）                   （弱耦合）
```

#### 2. 提高灵活性

- 子系统内部可以自由变化
- 只要Facade接口不变，客户端无需修改

#### 3. 提高安全性

- 不想让客户端访问的接口可以不暴露
- 只需要在Facade上开通必要的方法

#### 4. 符合高扇入原则

> "一个模块被多个其他模块共用 → 重用性高、通用性强"

---

### 三、FACADE模式的缺点

#### 不符合开闭原则

> "一旦在系统投产后发现有一个小错误，你怎么解决？完全遵从开闭原则，根本没办法解决。"

**风险**：
- 门面包含业务逻辑，违反单一职责
- 门面方法过多，成为"上帝对象"
- 子系统无法直接访问，丧失灵活性

---

### 四、MEDIATOR模式（中介者模式）

#### 模式定义

> "用一个中介对象来封装一系列的对象交互。中介者使各对象不需要显式地相互引用，从而使其耦合松散，而且可以独立地改变它们之间的交互。"

#### 问题背景

对象之间直接通信会导致：
- 对象间形成网状耦合
- 一个对象变化影响多个其他对象
- 系统难以维护和扩展

```
无Mediator（网状耦合）：
  A ──→ B
  │╲    │╲
  ▼ ▼   ▼ ▼
  C ──→ D

有Mediator（星型结构）：
      ┌─────┐
      │Medi-│
      │ator │
      └──┬──┘
    ┌────┼────┐
    ▼    ▼    ▼
    A     B    C
```

#### 模式结构

```
┌─────────────────────────┐
│      Mediator           │  中介者接口
│  notify()               │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│   ConcreteMediator      │  具体中介者
│  colleagueA.op()       │
│  colleagueB.op()        │
└────────────┬────────────┘
             │
    ┌────────┴────────┐
    ▼                 ▼
┌─────────┐      ┌─────────┐
│Colleague│      │Colleague│
│    A    │      │    B    │
└─────────┘      └─────────┘
```

#### 代码示例：聊天室

**没有Mediator**：
```java
// 用户之间直接通信
class User {
    void sendMessage(String to, String msg) {
        // 直接引用另一个用户
    }
    void receiveMessage(String msg) { }
}

// 问题：每个用户都要知道所有其他用户
```

**有Mediator**：
```java
interface Mediator {
    void sendMessage(User from, String to, String msg);
}

class ChatRoom implements Mediator {
    private Map<String, User> users;

    public void sendMessage(User from, String toName, String msg) {
        User to = users.get(toName);
        if (to != null) {
            to.receiveMessage(from.getName() + ": " + msg);
        }
    }
}

class User {
    private Mediator chatRoom;
    private String name;

    public void send(String to, String msg) {
        chatRoom.sendMessage(this, to, msg);
    }

    public void receive(String msg) {
        System.out.println(name + " received: " + msg);
    }
}
```

### 五、FACADE vs MEDIATOR

| 对比 | FACADE | MEDIATOR |
|-----|--------|----------|
| 目的 | 简化接口 | 解耦对象 |
| 调用方向 | 客户端调用门面 | 对象之间通过中介者通信 |
| 知识 | Facade不知道子系统细节 | Mediator知道所有对象 |
| 行为 | 被动（被调用） | 主动（协调） |

---

### 六、模式的使用原则

#### 何时使用FACADE

1. 需要一个复杂子系统的简单接口
2. 子系统相对独立
3. 预防低水平人员带来的风险扩散

#### 何时使用MEDIATOR

1. 对象间存在复杂的引用关系
2. 想自定义一个或多个类中的行为，但不想生成子类
3. 存在适应多步操作的可复用行为

#### 模式误用警示

> "一个子系统可以有多个门面" —— 但不要创建"上帝门面"

> "门面不参与子系统内的业务逻辑" —— 门面只是委托，不应包含实际业务

## 第十三至十五章核心总结

### 一、模式分类

| 章节 | 模式 | 类型 | 核心机制 |
|-----|------|-----|---------|
| 第十三章 | COMMAND | 行为型 | 封装请求 |
| 第十三章 | ACTIVE OBJECT | 并发型 | 异步执行 |
| 第十四章 | TEMPLATE METHOD | 行为型 | 继承 |
| 第十四章 | STRATEGY | 行为型 | 委托 |
| 第十五章 | FACADE | 结构型 | 简化接口 |
| 第十五章 | MEDIATOR | 行为型 | 解耦对象 |

---

### 二、模式对比速查

| 选择困惑 | 结论 |
|---------|------|
| Template Method vs Strategy | 继承 vs 委托；需要动态切换选Strategy |
| FACADE vs MEDIATOR | 简化接口 vs 解耦对象；门面是被动的 |

---

### 三、关键洞见

> "COMMAND模式的核心非常简单，但当你使用它的时候，它就会闪现光华。"

> "Strategy模式完全遵循DIP原则，允许每个具体实现都可以被多个不同的通用算法操纵。"

> "模式不是万能药。没有遇到变化的时候，就用模式，当然会变得臃肿了。"

> "门面模式注重'统一的对象'，提供一个访问子系统的唯一通道。"

> "门面不参与子系统内的业务逻辑，否则会违反单一职责。"

---

### 四、设计原则检查

| 场景 | 应遵循原则 |
|-----|-----------|
| 命令封装 | COMMAND模式 |
| 异步执行 | ACTIVE OBJECT模式 |
| 算法骨架不变，步骤变化 | TEMPLATE METHOD模式 |
| 算法需要动态切换 | STRATEGY模式 |
| 简化复杂子系统接口 | FACADE模式 |
| 解耦对象间交互 | MEDIATOR模式 |

---

### 五、模式应用决策树

```
需要模式吗？
    │
    ├── 变化还没有发生？ → YAGNI，不要用模式
    │
    └── 确实需要模式？
            │
            ├── 封装请求？ → COMMAND
            ├── 异步执行？ → ACTIVE OBJECT
            ├── 算法骨架固定，步骤变化？ → TEMPLATE METHOD
            ├── 算法需要切换？ → STRATEGY
            ├── 简化复杂接口？ → FACADE
            └── 对象耦合严重？ → MEDIATOR
```
