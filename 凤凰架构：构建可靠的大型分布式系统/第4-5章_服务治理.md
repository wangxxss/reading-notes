# 第4-5章：服务治理

## 本章要点

### 第4章：服务治理框架

#### 4.1 什么是服务治理
服务治理是**管理分布式系统中服务运行状态**的实践活动，包括：
- 服务注册与发现
- 服务监控与告警
- 服务配置管理
- 服务生命周期管理

#### 4.2 主流服务治理框架对比

| 框架 | 开发语言 | 特点 | 适用场景 |
|------|---------|------|---------|
| **Spring Cloud Netflix** | Java | 生态完善Netflix OSS | Java微服务 |
| **Spring Cloud Alibaba** | Java | 阿里开源，兼容Dubbo | 国内Java项目 |
| **Apache Dubbo** | Java | 高性能RPC，服务治理 | 高并发场景 |
| **Kratos** | Go | 哔哩哔哩开源 | Go微服务 |
| **Spring Cloud** | Java | 通用框架 | 跨语言困难 |

#### 4.3 Spring Cloud生态详解
- **Eureka**：服务注册与发现（已停止维护）
- **Nacos**：阿里开源，服务发现+配置管理+服务管理
- **Sentinel**：流量控制、熔断降级
- **Seata**：分布式事务解决方案
- **Gateway/Zuul**：API网关

#### 4.5 API网关设计模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **路由模式** | 根据路径/Header将请求转发到后端服务 | 基本的服务路由 |
| **聚合模式** | 将多个后端服务的结果聚合为一个响应 | 减少前端请求次数 |
| **BFF模式** | 为不同客户端（Web/App/小程序）提供专属网关 | 多端适配 |

- **网关核心功能**：路由转发、协议转换、限流熔断、认证鉴权、日志监控
- **主流网关**：Kong（基于Nginx+Lua）、APISIX（Apache）、Spring Cloud Gateway（Java）、Envoy（云原生）

#### 4.4 配置中心
- **为什么需要配置中心**：
  - 配置分散在不同服务中，难以统一管理
  - 配置变更需要重启服务
  - 环境差异（dev/test/prod）配置不同

- **配置中心核心功能**：
  - 配置的**增删改查**
  - 配置的**版本管理**
  - 配置的**热更新**（不重启生效）
  - 配置的**灰度发布**
  - 配置的**权限管理**

- **主流配置中心**：Nacos、Apollo、Spring Cloud Config、Consul

### 第5章：服务安全

#### 5.1 身份认证与授权
- **认证（Authentication）**：验证"你是谁"
- **授权（Authorization）**：验证"你能做什么"

#### 5.2 认证授权方案

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **Session-Cookie** | 服务端存储会话状态 | 传统Web应用 |
| **JWT** | 无状态令牌 | 微服务、跨域场景 |
| **OAuth2.0** | 授权协议 | 第三方登录、开放API |
| **LDAP** | 目录服务认证 | 企业内部系统 |

#### 5.3 JWT（JSON Web Token）
```
JWT = Header.Payload.Signature

Header: {"alg": "HS256", "typ": "JWT"}
Payload: {"sub": "123456", "name": "John", "exp": 1699999999}
Signature: HMAC-SHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)
```

- **JWT优点**：
  - 无状态，可扩展
  - 跨语言，支持多语言SDK
  - 自包含，包含用户信息

- **JWT缺点**：
  - 无法主动注销/过期（解决：Token吊销策略）
  - 令牌体积大
  - 秘钥管理复杂

#### 5.4 JWT Token吊销策略

JWT的核心问题是**无状态导致无法主动失效**。以下是常见解决方案：

##### 黑名单模式（Token Blacklist）
```
用户注销 → 将token加入黑名单 → 验证时检查黑名单
```
- **实现**：将已吊销的JTI（JWT ID）存入Redis
- **优点**：实现简单，立即生效
- **缺点**：失去了JWT的无状态优势，增加了存储依赖

##### 白名单模式（Token Whitelist）
```
用户登录 → 将token ID存入白名单 → 验证时检查白名单
```
- **实现**：将有效token的JTI存入Redis，验证时必须存在白名单中
- **优点**：支持精细化控制，可主动失效任意token
- **缺点**：完全依赖存储，失去了JWT的无状态优势

##### 短过期 + 刷新令牌（Short Expiry + Refresh Token）
```
Access Token：短过期（15min）
Refresh Token：长过期（7day），存储在服务端
```
- **流程**：
  1. Access Token过期后，客户端使用Refresh Token换取新的Access Token
  2. Refresh Token一次性使用（颁发新Token时吊销旧Token）
  3. 用户注销时，吊销Refresh Token
- **优点**：平衡了安全性和无状态性
- **缺点**：实现复杂度增加

##### 渐进式过期（SlidingExpiration）
```
每次请求都刷新过期时间
→ 30min无活动则过期
→ 有活动则最多延长到60min
```
- **优点**：用户体验好，自动续期
- **缺点**：实现复杂，不适合高安全场景

##### 组合策略建议

| 场景 | 策略 | 说明 |
|------|------|------|
| 普通应用 | 短过期 + Refresh Token | 平衡安全与实现复杂度 |
| 高安全场景 | 短过期 + Refresh Token + 黑名单 | Access Token被窃取可快速吊销 |
| 简单场景 | 仅短过期 | 依赖过期时间自然失效 |
| 登出即失效 | Refresh Token白名单 | 登出时删除Refresh Token |

##### JTI（JWT ID）的重要性
- JTI是Token的唯一标识，用于支持吊销功能
- 建议在Payload中包含：
```json
{
  "sub": "user123",
  "jti": "unique-token-id-uuid",  // 用于吊销
  "iat": 1699999999,
  "exp": 1699999999
}
```

#### 5.5 OAuth2.0 四种授权模式
1. **授权码模式**：最安全，适合有后端的应用
2. **简化模式**：适合纯前端应用
3. **密码模式**：信任客户端，直接给用户名密码
4. **客户端模式**：机器对机器授权

#### 5.6 服务间认证
- **MTLS（双向TLS）**：客户端和服务端互相验证证书
- **服务账号/Token**：使用服务账号的Token进行认证
- **API Key**：简单场景下的身份标识

#### 5.7 访问控制模型
- **RBAC（基于角色的访问控制）**：用户→角色→权限
- **ABAC（基于属性的访问控制）**：基于用户属性、资源属性、环境属性
- **零信任安全**：永不信任，始终验证
  - **核心原则**：
    1. **最小权限**：只授予完成任务所需的最小权限
    2. **持续验证**：每次访问都要验证身份和权限，不依赖网络位置
    3. **假设被突破**：设计时假设内网已被攻破，最小化爆炸半径
  - **实践手段**：微分段（Micro-Segmentation）、MTLS、身份感知代理（IAP）

## 关键知识点

1. **Nacos vs Eureka**：
   - Eureka 1.x已停止维护，Eureka 2.x不开源
   - Nacos支持CP和AP两种模式，功能更全面
   - 生产环境推荐使用Nacos

2. **JWT vs Session**：
   - Session需要分布式存储（Redis），适合有后端的应用
   - JWT无状态，适合前端分离和微服务架构

3. **OAuth2 vs JWT**：
   - OAuth2是授权协议，JWT是令牌格式
   - OAuth2可以使用JWT作为令牌

## 配置中心选择建议

```
Apollo:
  + 功能完善，企业级
  - 部署复杂，需要MySQL

Nacos:
  + 部署简单，内嵌数据库
  + 同时支持注册中心和配置中心
  - 相对年轻

Spring Cloud Config:
  + Spring生态原生
  - 需要配合Git使用，功能相对简单
```

## 本章小结
服务治理是分布式系统稳定运行的关键保障。配置中心实现配置的集中管理和热更新，安全机制保障系统的身份认证和访问控制。学习本章时，建议结合实际项目理解各框架的选型。
