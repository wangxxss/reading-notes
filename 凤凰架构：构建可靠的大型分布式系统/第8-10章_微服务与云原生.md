# 第8-10章：微服务与云原生

## 本章要点

### 第8章：微服务架构

#### 8.1 微服务定义
- **微服务是一种架构风格**，将大型应用拆分为多个小型、自治的服务
- 每个服务运行在独立进程中，有自己的数据库
- 服务可通过轻量级通信机制（HTTP API、消息队列）交互

#### 8.2 微服务特征
| 特征 | 说明 |
|------|------|
| **单一职责** | 每个服务只负责一个业务领域 |
| **轻量级通信** | 使用HTTP、gRPC、消息队列 |
| **独立性** | 独立部署、独立扩展、独立发布 |
| **去中心化** | 技术栈、数据存储去中心化 |
| **弹性设计** | 容错、降级、隔离 |

#### 8.3 微服务拆分策略

##### 按业务能力拆分
- 识别企业核心业务能力
- 每个能力对应一个服务
- 适合业务边界清晰的场景

##### 按领域驱动设计（DDD）拆分
- 核心域、支撑域、通用域
- 限界上下文（Bounded Context）
- 上下文映射（Context Mapping）

##### 拆分原则
1. **高内聚低耦合**：服务内高内聚，服务间低耦合
2. **独立演进**：服务可独立演进，不影响其他服务
3. **团队自治**：按团队划分服务边界
4. **2 Pizza原则**：团队规模2张披萨能吃饱（6-10人）

#### 8.4 微服务反模式
- **泥球模式（Ball of Mud）**：服务间过度依赖
- **共享数据库**：违反服务自治
- **分布式单体**：微服务外表，单体内核
- **过度拆分**：服务粒度过细

#### 8.5 服务网格（Service Mesh）
- **Sidecar模式**：每个服务实例伴随一个代理
- **数据平面**：Envoy、Istio代理
- **控制平面**：Istiod、Control Plane
- **优点**：业务代码与治理逻辑解耦、跨语言统一治理、流量管理精细化
- **缺点**：
  - 性能开销：每次调用多经过两次 Sidecar 代理，增加延迟（通常 1-3ms）
  - 运维复杂度：需要维护 Sidecar 的升级、配置、资源分配
  - 调试困难：链路经过代理层，排查问题更复杂
  - 资源消耗：每个 Pod 额外消耗 CPU 和内存

│  Service A  │         │  Service B  │
│ ┌─────────┐ │         │ ┌─────────┐ │
│ │ Sidecar │ │◄───────►│ │ Sidecar │ │
│ └─────────┘ │  网络   │ └─────────┘ │

       ▲                       ▲

### 第9章：云原生架构

#### 9.1 云原生定义
> 云原生是一套技术体系和方法论，充分利用云计算平台的优势构建和运行应用。

#### 9.2 12-Factor App 方法论
云原生应用的设计准则（由 Heroku 提出）：

| 因素 | 说明 |
|------|------|
| 1. 基准代码 | 一份基准代码，多份部署 |
| 2. 依赖 | 显式声明依赖关系 |
| 3. 配置 | 在环境中存储配置 |
| 4. 后端服务 | 把后端服务当作附加资源 |
| 5. 构建/发布/运行 | 严格分离构建和运行阶段 |
| 6. 进程 | 以无状态进程运行应用 |
| 7. 端口绑定 | 通过端口绑定提供服务 |
| 8. 并发 | 通过进程模型进行扩展 |
| 9. 易处理 | 快速启动和优雅终止 |
| 10. 开发/生产等价 | 保持开发、预发布、生产环境一致 |
| 11. 日志 | 把日志当作事件流 |
| 12. 管理进程 | 后台管理任务作为一次性进程运行 |

#### 9.3 云原生核心要素
| 要素 | 说明 |
|------|------|
| **容器** | 标准化封装，应用及其依赖 |
| **微服务** | 拆分后的细小服务 |
| **声明式API** | 描述期望状态而非过程 |
| **服务网格** | 流量管理、安全、可观测性 |
| **不可变基础设施** | 镜像即环境，环境不可变 |
| **DevOps** | 开发运维一体化 |

#### 9.4 Kubernetes核心概念
- **Pod**：最小调度单元，可以包含多个容器
- **Service**：服务发现和负载均衡
- **Deployment**：声明式更新，无状态应用
- **StatefulSet**：有状态应用
- **ConfigMap/Secret**：配置管理
- **Namespace**：资源隔离

#### 9.6 K8s调度与弹性机制
- **资源管理**：
  - `requests`：容器运行所需的最低资源保证
  - `limits`：容器可使用的资源上限
- **HPA（Horizontal Pod Autoscaler）**：根据 CPU/内存/自定义指标自动水平扩缩容
- **VPA（Vertical Pod Autoscaler）**：自动调整 Pod 的资源请求和限制
- **亲和性调度**：
  - 节点亲和性（nodeAffinity）：Pod 倾向调度到特定节点
  - Pod 亲和性（podAffinity）：Pod 倾向与特定 Pod 部署在一起
  - Pod 反亲和性（podAntiAffinity）：Pod 避免与特定 Pod 部署在一起（用于高可用分散部署）

#### 9.7 Operator模式：Kubernetes的扩展机制

**Operator**是Kubernetes的扩展机制，通过自定义资源（CRD）和自定义控制器来封装运维人员的手动操作，实现复杂应用的自动化管理。

##### 核心概念

| 概念 | 说明 |
|------|------|
| **CRD（Custom Resource Definition）** | 扩展Kubernetes API，定义新的资源类型 |
| **Controller** | 持续监控集群状态，确保实际状态匹配期望状态 |
| **Operator** | CRD + Controller的组合，封装复杂应用的运维逻辑 |
| **Reconciliation Loop** | 控制器循环：获取状态 → 比较差异 → 执行操作 → 重复 |

##### 工作原理

```
┌─────────────────────────────────────────────────────┐
│                  Operator 工作流程                    │
├─────────────────────────────────────────────────────┤
│                                                      │
│  用户创建 MyApp CR（Custom Resource）               │
│                    │                                │
│                    ▼                                │
│  ┌──────────────────────────────────────┐          │
│  │         API Server 存储              │          │
│  │    (MyApp CRD 实例)                  │          │
│  └───────────────┬──────────────────────┘          │
│                  │ Watch                            │
│                  ▼                                  │
│  ┌──────────────────────────────────────┐          │
│  │     Operator Controller               │          │
│  │  ┌────────────────────────────────┐ │          │
│  │  │ Reconciliation Loop             │ │          │
│  │  │ 1. 获取期望状态（从CR读取）      │ │          │
│  │  │ 2. 获取实际状态（查询集群）      │ │          │
│  │  │ 3. 比较差异                      │ │          │
│  │  │ 4. 执行操作使实际=期望           │ │          │
│  │  └────────────────────────────────┘ │          │
│  └───────────────┬──────────────────────┘          │
│                  │ 创建/更新资源                     │
│                  ▼                                  │
│  ┌──────────────────────────────────────┐          │
│  │     Kubernetes 原生资源              │          │
│  │  Deployment / StatefulSet / Service  │          │
│  └──────────────────────────────────────┘          │
│                                                      │
└─────────────────────────────────────────────────────┘
```

##### 知名Operator示例

| Operator | 厂商 | 功能 |
|----------|------|------|
| **Prometheus Operator** | CoreOS/CNCF | 自动化部署和配置Prometheus监控 |
| **Istio Operator** | Istio | 自动化部署和配置服务网格 |
| **cert-manager** | Jetstack | 自动化管理TLS证书 |
| **ExternalDNS** | Kubernetes SIG | 同步Service到DNS提供商 |
| **PostgreSQL Operator** | CrunchyData | 管理PostgreSQL集群 |
| **MongoDB Operator** | MongoDB | 管理MongoDB集群 |
| **TiDB Operator** | PingCAP | 管理TiDB集群 |

##### Operator开发框架

| 框架 | 语言 | 特点 |
|------|------|------|
| **Kubebuilder** | Go | 官方标准，生成CRD和Controller脚手架 |
| **Operator SDK** | Go/Ansible/Helm | 支持多种开发方式 |
| ** Kopf（Kubernetes Operator Pythonic Framework）** | Python | Python开发Operator的简单框架 |

##### Operator最佳实践

1. **错误处理**：使用指数退避重试，避免无效循环
2. **状态管理**：维护CR的`.status`字段反映实际状态
3. **Finalizer**：确保资源删除前执行清理逻辑
4. **Owner Reference**：设置资源的所有者，方便级联删除
5. **并发控制**：使用工作队列避免并发冲突
6. **充分测试**：使用envtest进行单元测试，集成测试覆盖升级/降级场景

##### Operator vs Helm vs 原生资源

| 维度 | 原生资源 | Helm | Operator |
|------|----------|------|----------|
| **复杂度** | 低 | 中 | 高 |
| **状态管理** | 无 | 部分 | 完整 |
| **自愈能力** | 无 | 无 | 有 |
| **适用场景** | 简单应用 | 一次性部署 | 有状态/复杂应用 |
| **升级能力** | 手动 | 部分支持 | 自动化 |

### 第10章：容器与Docker

#### 10.1 容器 vs 虚拟机
| 特性 | 容器 | 虚拟机 |
|------|------|--------|
| 启动速度 | 秒级 | 分钟级 |
| 资源占用 | 小（MB级） | 大（GB级） |
| 隔离性 | 进程级 | 硬件级 |
| 性能 | 近宿主 | 有损耗 |
| 生态 | 云原生标配 | 传统IT |

#### 10.2 Docker核心概念
- **镜像（Image）**：只读模板
- **容器（Container）**：镜像的运行实例
- **仓库（Registry）**：存储和分发镜像
- **Dockerfile**：构建镜像的脚本

#### 10.3 容器网络模式
- **bridge**：默认模式，容器通过docker0网桥通信
- **host**：容器与宿主机共享网络
- **overlay**：跨主机容器网络（Docker Swarm）
- **none**：禁用网络

#### 10.4 容器存储
- **Volume**：宿主机文件系统映射
- **Bind Mount**：指定宿主机路径
- **tmpfs Mount**：内存文件系统
- **NFS/CIFS**：网络存储

#### 10.5 容器编排
- **Docker Compose**：单机多容器编排
- **Kubernetes**：生产级容器编排
- **Helm**：Kubernetes包管理器

#### 10.6 镜像优化
- 选用小型基础镜像（Alpine、scratch）
- 多阶段构建
- 减少层数
- 合并RUN指令
- 使用.dockerignore

## 本章小结
微服务是架构演进的结果，云原生是微服务的最佳部署平台。容器技术（Docker）和容器编排（Kubernetes）已成为现代分布式系统的基础设施。理解这些技术的设计理念，对于构建可靠的分布式系统至关重要。
