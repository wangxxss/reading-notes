# 第6-7章：容错与高可用

## 本章要点

### 第6章：高可用架构

#### 6.1 可用性的度量
- **可用性 = MTBF / (MTBF + MTTR)**
  - MTBF（Mean Time Between Failures）：平均故障间隔
  - MTTR（Mean Time To Repair）：平均修复时间

- **提高可用性的本质**：减少MTTR，延长MTBF

#### 6.2 高可用架构设计原则
1. **消除单点故障**：关键组件必须有冗余
2. **故障隔离**：将故障控制在最小范围
3. **快速失败**：不要让故障扩散
4. **优雅降级**：牺牲部分功能保证核心功能
5. **可恢复性**：故障后能自动或手动恢复

#### 6.3 常见高可用架构模式

##### 主备模式（Active-Standby）
```
       ┌─────────┐
       │  请求   │
       └────┬────┘
            │
    ┌───────┴───────┐
    ▼               ▼
┌────────┐     ┌────────┐
│ Active │     │Standby │
│ 主节点  │     │ 备节点  │
└────────┘     └────────┘
```
- 主机故障时，备机接管
- 切换方式：手动切换、自动切换

##### 主主模式（Active-Active）
- 两个节点都承担流量
- 优点：资源利用率高
- 缺点：数据同步复杂

##### 集群模式
- **奇数节点**：3节点、5节点、7节点
- **多数派原则**：n/2+1才能做出决策
- **脑裂问题**：网络分区导致多个节点认为自己是主节点

#### 6.4 故障检测机制
- **心跳检测**：定期发送心跳包，超时未收到则判定故障
  - 固定超时：简单但误判率高（网络抖动可能触发误判）
  - **Phi Accrual Failure Detector（自适应故障检测）**：
    - 根据历史心跳间隔动态计算"怀疑程度"（φ值）
    - φ值越高，节点故障的概率越大
    - 自适应网络波动，减少误判
    - Cassandra、Akka 等系统采用此算法

#### 6.5 CAP定理
- **C（Consistency）**：一致性
- **A（Availability）**：可用性
- **P（Partition Tolerance）**：分区容错性

> 分布式系统只能同时满足其中两个，无法同时满足全部三个。

| 组合 | 说明 | 代表系统 |
|------|------|---------|
| CA | 放弃分区容错，不推荐 | 单机数据库 |
| CP | 放弃可用性 | Zookeeper、HBase |
| AP | 放弃一致性 | Eureka、Cassandra |

#### 6.6 PACELC理论（CAP的扩展）
- CAP只讨论了**有网络分区时**的权衡，PACELC补充了**无分区时**的权衡：
  - **P**artition时选择 **A**vailability 还是 **C**onsistency
  - **E**lse（无分区时）选择 **L**atency 还是 **C**onsistency

- **PACELC公式**：`Latency vs Consistency (when no partition)`

- 示例：
  - **Cassandra**：PA/EL（分区时选可用性，无分区时选低延迟）
  - **HBase**：PC/EC（分区时选一致性，无分区时也选一致性）
  - **DynamoDB**：PA/EL（默认最终一致，可选强一致读）
  - **Zookeeper**：PC/EC（始终选择一致性）

- **实践意义**：PACELC帮助我们理解为什么某些系统在正常运行时响应快，某些系统始终保持强一致。在选型时需要考虑：你的业务是延迟敏感还是一致性敏感？

#### 6.7 BASE理论
- **Basically Available**：基本可用
- **Soft State**：软状态
- **Eventually Consistent**：最终一致性

> BASE是对CAP中一致性和可用性权衡的总结，是大规模分布式系统的事实标准。

### 第7章：容错设计

#### 7.1 故障类型
| 故障类型 | 说明 | 示例 |
|---------|------|------|
| **瞬时故障** | 短暂出现后可自愈 | 网络抖动 |
| **临时故障** | 需要一定时间恢复 | 服务重启 |
| **永久故障** | 必须人工干预 | 硬盘损坏 |

#### 7.2 重试机制
- **何时重试**：网络超时、服务暂时不可用
- **何时不能重试**：业务逻辑错误、数据问题

- **重试策略**：
  - 固定间隔
  - 指数退避（Exponential Backoff）
  - 抖动（Jitter）防止惊群

```
重试间隔 = min(最大间隔, 基础间隔 × 2^重试次数) + 随机抖动
```

#### 7.3 超时与熔断
- **超时设置**：太长影响响应，太短误判
- **熔断器状态机**：
```
           ┌─────────┐
           │ Closed  │ ◄──── 探测成功 ────┐
           │ 关闭态  │                    │
           └────┬────┘                    │
                │ 失败超阈值              │
                ▼                        │
           ┌─────────┐    超时后探测   ┌──────────┐
           │  Open   │ ─────────────▶ │Half-Open │
           │ 打开态  │               │ 半开态   │
           └─────────┘               └────┬─────┘
                ▲                         │ 探测失败
                └─────────────────────────┘
```
- **状态转换说明**：
  - Closed → Open：失败次数/比例超过阈值
  - Open → Half-Open：经过设定的超时时间后，允许少量探测请求通过
  - Half-Open → Closed：探测请求成功，恢复正常
  - Half-Open → Open：探测请求仍然失败，继续熔断

#### 7.4 幂等性设计
- **什么是幂等**：多次执行结果与一次执行结果相同
- **为什么重要**：重试机制需要幂等保障

- **实现方式**：
  - 数据库唯一索引
  - 乐观锁/悲观锁
  - 去重表
  - 状态机
  - 分布式锁

#### 7.5 补偿机制
- **正向补偿**：A→B→C，失败时C→B→A回滚
- **逆向补偿**：记录操作，失败时执行反向操作
- **异步补偿**：通过消息队列解耦，异步执行补偿

#### 7.6 隔离设计
- **线程隔离**：不同任务使用不同线程池
- **进程隔离**：不同服务使用不同进程
- **容器隔离**：使用容器技术隔离资源
- **集群隔离**：核心服务与非核心服务分开部署
- **机房隔离**：多机房部署

#### 7.7 限流设计
- **限流算法对比**：

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 计数器 | 简单 | 有临界问题 | 简单限流 |
| 滑动窗口 | 精确 | 实现复杂 | 精确限流 |
| 漏桶 | 流量平滑 | 不能应对突发 | 稳定流量 |
| 令牌桶 | 允许突发 | - | 突发流量 |

#### 7.8 混沌工程（Chaos Engineering）
- **定义**：通过主动向系统注入故障，验证系统在异常情况下的韧性
- **核心原则**：
  1. 建立系统正常行为的基线
  2. 假设正常行为在实验组和对照组中都持续
  3. 引入真实世界中的故障变量（网络延迟、节点宕机、磁盘满等）
  4. 观察实验组与对照组的差异
- **主流工具**：
  - **Chaos Monkey**（Netflix）：随机终止生产环境的虚拟机实例
  - **ChaosBlade**（阿里）：支持多种故障注入场景
  - **Litmus**（CNCF）：Kubernetes 原生的混沌工程平台
- **实践建议**：先从测试环境开始，逐步推向生产环境；从简单故障开始，逐步增加复杂度

## 高可用设计检查清单

```
□ 服务无单点故障
□ 关键数据有备份
□ 流量有兜底方案
□ 故障能快速检测
□ 故障能自动转移
□ 降级策略明确
□ 恢复流程清晰
□ 监控告警到位
```

## 本章小结
容错与高可用是分布式系统设计的核心话题。理解CAP定理、BASE理论、重试熔断、幂等补偿等概念，是构建可靠分布式系统的基础。在实际工作中，需要根据业务场景权衡可用性和一致性。
