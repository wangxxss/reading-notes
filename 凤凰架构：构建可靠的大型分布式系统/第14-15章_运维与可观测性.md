# 第14-15章：运维与可观测性

## 本章要点

### 第14章： DevOps与持续交付

#### 14.1 DevOps文化
- **DevOps = Development + Operations**
- 核心理念：开发与运维一体化，缩短交付周期，提高质量

#### 14.2 持续集成（CI）
- **Continuous Integration**：频繁提交代码，自动构建和测试
- 核心实践：
  - 代码频繁合并
  - 自动构建
  - 自动测试
  - 快速反馈

#### 14.3 持续交付（CD）
- **Continuous Delivery**：代码随时可发布
- 核心实践：
  - 自动化部署
  - 部署流水线
  - 环境一致性
  - 特性开关

#### 14.4 持续部署
- **Continuous Deployment**：代码自动发布到生产环境
- 比持续交付更进一步
- 需要完善的测试和监控体系

#### 14.5 部署策略
| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **蓝绿部署** | 两套环境，切换流量 | 快速回滚 | 资源占用大 |
| **金丝雀发布** | 逐步放量 | 风险可控 | 周期长 |
| **滚动发布** | 逐个替换实例 | 资源利用率高 | 回滚慢 |
| **功能开关** | 通过配置控制功能 | 无需重新部署 | 技术债 |

#### 14.6 流水线设计
```
代码提交 → 编译构建 → 单元测试 → 集成测试 → 镜像构建 → 部署测试环境
    → 自动验收测试 → 部署预发布 → 部署生产 → 监控验证
```

#### 14.7 基础设施即代码（IaC）
- 使用代码管理基础设施
- 版本控制、可重复、可测试
- **工具**：Terraform、Pulumi、Ansible

#### 14.8 GitOps
- **定义**：以 Git 仓库作为基础设施和应用配置的**唯一事实来源**
- **核心原则**：
  - 声明式描述所有环境配置
  - 所有变更通过 Git PR 审批
  - 自动化 Agent 将 Git 状态同步到集群
  - 偏差自动检测和修复
- **主流工具**：
  - **ArgoCD**：Kubernetes 原生的 GitOps CD 工具
  - **Flux**：CNCF 项目，轻量级 GitOps 工具
- **vs 传统 CI/CD**：传统是"推"模式（CI 推送部署），GitOps 是"拉"模式（Agent 从 Git 拉取期望状态）

### 第15章：可观测性

#### 15.1 可观测性定义
> 可观测性是指系统能够从外部输出推断其内部状态的能力。

#### 15.2 可观测性三支柱
| 支柱 | 说明 | 解决的问题 |
|------|------|-----------|
| **Metrics（指标）** | 数值型的聚合数据 | 系统健康状态、性能趋势 |
| **Logs（日志）** | 离散的事件记录 | 问题的详细信息 |
| **Traces（链路追踪）** | 请求在系统中的完整路径 | 分布式调用链分析 |

#### 15.3 Metrics（指标）

##### 指标类型
| 类型 | 说明 | 示例 |
|------|------|------|
| **Gauge** | 当前值，可增可减 | CPU使用率、内存使用量 |
| **Counter** | 只增不减的计数器 | 请求总数、错误总数 |
| **Histogram** | 统计分布 | 响应时间分布 |
| **Summary** | 客户端计算的分位数 | P99响应时间 |

##### 指标采集框架
- **Prometheus**：CNCF项目，拉取模式
- **InfluxDB**：时序数据库
- **StatsD**：聚合统计指标

##### 关键业务指标
```
系统指标：CPU、内存、磁盘、网络
应用指标：QPS、RT、错误率、并发数
业务指标：订单量、转化率、DAU
```

#### 15.4 Logs（日志）

##### 日志级别
| 级别 | 优先级 | 使用场景 |
|------|--------|---------|
| **TRACE** | 最低 | 详细调试信息 |
| **DEBUG** | 低 | 开发调试 |
| **INFO** | 中 | 一般信息 |
| **WARN** | 高 | 警告信息 |
| **ERROR** | 错误 | 错误信息 |
| **FATAL** | 最高 | 致命错误 |

##### 日志规范
- **结构化日志**：JSON格式，便于分析
- **日志关联ID**：TraceID、RequestID
- **日志保留策略**：根据存储成本和合规要求

##### 日志采集架构
```
应用日志 → 日志代理(Fluentd/Filebeat) → 消息队列(Kafka) → 日志存储(ES) → 可视化(Kibana)
```

##### 主流日志系统
- **ELK Stack**：Elasticsearch + Logstash + Kibana
- **Loki**：Grafana开源，与Prometheus集成
- **EFK**：Elasticsearch + Fluentd + Kibana

#### 15.5 Traces（链路追踪）

##### 链路追踪概念
- **Span**：基本工作单元，一次操作
- **Trace**：一次请求的完整调用链
- **Parent Span**：调用关系

```
Trace: 整个请求生命周期
├── Span A (Root)
│   ├── Span B (A的子span)
│   │   └── Span D (B的子span)
│   └── Span C (A的子span)
```

##### OpenTelemetry（OTel）

**背景**：OpenTracing（Google）和 OpenCensus（Microsoft）两大标准合并为 OpenTelemetry，成为可观测性的统一标准。

**核心特性**：
- **三合一**：Metrics + Logs + Traces 统一采集标准
- **厂商中立**：一次埋点，可同时导出到多个后端（Jaeger、Prometheus、Datadog 等）
- **自动埋点**：支持 Java、Go、Python、Node.js 等多语言自动 instrumentation
- **OTLP 协议**：统一的数据传输协议

**架构组件**：

##### OpenTelemetry 集成实践

###### 1. OTel SDK 埋点

**自动埋点**（Auto Instrumentation）：
```yaml
# Java Agent 自动埋点示例
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.service.name=my-app \
     -Dotel.exporter.otlp.endpoint=http://collector:4317 \
     -jar my-app.jar
```

**手动埋点**（Manual Instrumentation）：
```go
// Go 手动埋点示例
func handleRequest(ctx context.Context, req Request) {
    ctx, span := tracer.Start(ctx, "handleRequest")
    defer span.End()

    span.SetAttributes("user.id", req.UserID)

    // 业务逻辑
    result := businessLogic(ctx, req)

    span.SetAttributes("result.count", len(result))
}
```

###### 2. OTel Collector 部署配置

OTel Collector 是数据采集、处理、导出的中间件：

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 1s
    send_batch_size: 1024
  memory_limiter:
    check_interval: 1s
    limit_percentage: 75

exporters:
  jaeger:
    endpoint: jaeger:14250
    tls:
      insecure: true
  prometheus:
    endpoint: "0.0.0.0:8889"

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, memory_limiter]
      exporters: [jaeger]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [prometheus]
```

###### 3. 数据关联与上下文传播

**TraceContext 传播**：
```java
// HTTP Headers 中的 TraceContext
// 将 TraceId 和 SpanId 传递到下游服务
headers.add("traceparent", "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01");
headers.add("tracestate", "congo=t61rcWkgMzE");
```

**Baggage 传递业务上下文**：
```go
// 设置 baggage
ctx = baggage.ContextWithValue(ctx, baggageKey("user-role"), "admin")

// 获取 baggage
role, _ := baggage.Value(ctx, baggageKey("user-role"))
```

###### 4. 采样策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **AlwaysOn** | 100% 采样 | 测试环境、调试阶段 |
| **AlwaysOff** | 0% 采样 | 性能敏感场景 |
| **Fixed Rate** | 固定比例采样 | 生产环境初步方案 |
| **Tail-based** | 请求结束后决定采样 | 需要完整尾部分布（延迟分析） |

```yaml
# Tail-based 采样配置示例
processors:
  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-policy
        type: status_code
        status_code: {status_codes: [ERROR]}
      - name: slow-traces-policy
        type: latency
        latency: {threshold_ms: 1000}
      - name: probabilistic-policy
        type: probabilistic
        probabilistic: {sampling_percentage: 10}
```

###### 5. 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **数据丢失** | Collector 内存不足 | 增加 memory_limiter processor |
| **延迟高** | 同步发送阻塞 | 使用 batch processor 异步发送 |
| **Trace 不完整** | 下游服务未传递 TraceContext | 确保所有 HTTP/gRPC 调用传播 headers |
| **性能开销大** | 采样率过高 | 调整采样策略，使用尾采样 |

#### 15.6 可观测性实践

##### 告警设计
- **告警等级**：P1（紧急）、P2（重要）、P3（一般）、P4（提示）
- **告警收敛**：减少告警风暴
- **告警抑制**：避免重复告警
- **on-call轮班**：7x24响应

##### 监控指标设计原则（USE方法）
- **Utilization**：利用率
- **Saturation**：饱和度
- **Errors**：错误

##### 监控指标设计原则（RED方法）
- **Rate**：请求率
- **Errors**：错误率
- **Duration**：响应时间

##### 四大黄金信号（Google SRE）
- **Latency（延迟）**：请求处理时间，需区分成功请求和失败请求的延迟
- **Traffic（流量）**：系统承受的请求量（如 HTTP QPS、网络 I/O）
- **Errors（错误）**：请求失败的比例
- **Saturation（饱和度）**：系统资源的利用程度，预警瓶颈

> USE 方法适合**基础设施**监控，RED 方法适合**微服务**监控，四大黄金信号是 SRE 的**综合视角**。

##### SLO/SLI/SLA
- **SLI（Service Level Indicator）**：服务水平指标
- **SLO（Service Level Objective）**：服务水平目标
- **SLA（Service Level Agreement）**：服务水平协议

```
SLA: 99.9% 可用性
SLO: 99.95% 可用性 (内部目标更高)
SLI: 当前可用性 99.97%
```

## 可观测性Checklist

```
□ 应用指标：QPS、RT、错误率、并发
□ 系统指标：CPU、内存、磁盘、网络
□ 业务指标：订单、用户行为
□ 日志规范：结构化、TraceID关联
```

## 本章小结
可观测性是保障系统稳定运行的基础。"指标+日志+链路追踪"三支柱构成了完整的可观测性体系。良好的可观测性能够帮助我们快速发现问题、定位问题、解决问题，是SRE和DevOps的核心实践。
