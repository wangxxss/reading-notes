# 第2-3章：分布式基础（通信与协作）

## 本章要点

### 第2章：分布式通信

#### 2.1 网络通信基础
- **OSI七层模型** vs **TCP/IP四层模型**
- 分布式系统中，网络是连接各个节点的**基础设施**
- 网络延迟是分布式系统性能瓶颈的重要来源之一

#### 2.2 远程调用的透明性谬误
- **分布式计算的八大谬误**（Fallacies of Distributed Computing）：
  1. 网络是可靠的
  2. 延迟是零
  3. 带宽是无限的
  4. 网络是安全的
  5. 拓扑不会变化
  6. 只有一个管理员
  7. 传输成本为零
  8. 网络是同质的
- **核心教训**：远程调用 ≠ 本地调用，必须考虑延迟、失败、重试等问题

#### 2.3 远程过程调用（RPC）
- RPC是分布式系统中最常见的**进程间通信**方式
- 核心思想：像调用本地函数一样调用远程函数

```
调用方 Client                    服务提供方 Server
    |                                  |
    |--- stub (代理) ----------------->|
    |                                  |--- 执行远程方法
    |<-- stub (代理) ------------------|
    |                                  |
```

- **主流RPC框架**：gRPC、Thrift、Dubbo、Feign

#### 2.4 REST vs RPC vs GraphQL

| 维度 | REST | RPC | GraphQL |
|------|------|-----|---------|
| **设计哲学** | 面向资源 | 面向动作 | 面向查询 |
| **传输协议** | HTTP | 多种（TCP/HTTP2） | HTTP |
| **数据格式** | JSON为主 | Protobuf/Thrift | JSON |
| **耦合度** | 低（标准HTTP语义） | 高（需共享接口定义） | 中 |
| **性能** | 中 | 高 | 中 |
| **适用场景** | 对外开放API | 内部服务间高性能调用 | 前端灵活查询、聚合多数据源 |
| **过度获取** | 常见 | 不常见 | 按需获取，无此问题 |

#### 2.5 序列化与反序列化
| 序列化方式 | 优点 | 缺点 |
|-----------|------|------|
| JSON | 跨语言、可读性好 | 体积大、速度慢 |
| Protobuf | 体积小、速度快 | 不可读、需要定义文件 |
| Avro | 支持schema演进 | 学习曲线陡峭 |
| Kryo | 高性能 | 跨语言支持差 |

#### 2.6 通信协议详解

##### HTTP/1.1
- **特性**：短连接、文本协议
- **问题**：每次请求需要建立TCP连接（连接复用困难），队头阻塞（HOL Blocking）
- **适用场景**：简单场景、兼容性要求高的环境

##### HTTP/2
- **核心特性**：
  - **多路复用（Multiplexing）**：一个TCP连接上并行多个请求/响应
  - **头部压缩（HPACK）**：减少头部冗余
  - **服务器推送（Server Push）**：服务端主动推送资源
  - **流优先级**：支持设置请求优先级
- **问题**：仍存在TCP层面的队头阻塞（一个TCP packet丢失会阻塞所有流）

##### HTTP/3 (QUIC)
- **核心改进**：
  - **基于UDP**：避免了TCP的队头阻塞问题
  - **QUIC协议**：内置流量控制、拥塞控制、连接迁移
  - **0-RTT/1-RTT**：减少握手延迟
  - **连接迁移**：网络切换时连接保持（如移动网络切换）

- **HTTP/3 vs HTTP/2 对比**：

| 特性 | HTTP/2 | HTTP/3 |
|------|--------|--------|
| 传输层 | TCP | UDP (QUIC) |
| 队头阻塞 | TCP层面存在 | 无（UDP无 HOL） |
| 连接建立 | 1-RTT | 1-RTT (首次) / 0-RTT (重连) |
| 连接迁移 | 需要新连接 | CID保持，连接不中断 |
| 前向纠错 | 无 | 有（减少重传） |
| 部署复杂度 | 低（兼容HTTP/2） | 高（需UDP穿透） |
| 浏览器支持 | 广泛 | 现代浏览器支持 |

##### gRPC
- **定位**：高性能、开源的RPC框架
- **技术栈**：HTTP/2 + Protobuf
- **核心优势**：
  - **二进制序列化**：Protobuf比JSON小6-10倍，解析快
  - **多语言支持**：几乎所有主流语言
  - **双向流**：支持客户端流、服务端流、双向流
  - **完整契约**：.proto文件定义服务和消息格式
- **适用场景**：微服务间通信、低延迟场景、移动端到后端

##### 协议选择建议

| 场景 | 推荐协议 | 原因 |
|------|---------|------|
| 浏览器 ↔ 服务端 | HTTP/2 / HTTP/3 | 标准化、广泛支持 |
| 微服务间调用 | gRPC | 高性能、强类型、双向流 |
| 对外开放API | REST + JSON | 易用、调试友好、生态好 |
| 移动端API | gRPC + HTTP/2 | 低延迟、省流量 |
| 需要流式处理 | gRPC Streaming | 原生支持流 |

### 第3章：分布式协作

#### 3.1 服务发现
- **为什么需要服务发现**：分布式系统中，服务实例的地址是动态变化的
- **核心功能**：注册、注销、订阅、查询

| 模式 | 代表技术 | 特点 |
|------|---------|------|
| **客户端发现** | Eureka、Zookeeper | 客户端直接查询注册中心 |
| **服务端发现** | Consul、Nginx | 客户端通过负载均衡器访问 |
| **DNS发现** | CoreDNS | 简单但功能有限 |

#### 3.2 负载均衡
- **负载均衡策略**：
  - 随机、加权随机
  - 轮询、加权轮询
  - 最少连接
  - IP哈希
  - 一致性哈希

- **负载均衡位置**：
  - 客户端负载均衡（如Ribbon）
  - 代理层负载均衡（如Nginx、Envoy）

#### 3.3 服务路由
- **灰度发布**：按比例/规则将流量切分到新版本
- **蓝绿部署**：两套环境，切换流量
- **金丝雀发布**：先小比例流量，逐步扩大

#### 3.4 服务限流
- **限流算法**：
  - 计数器算法：简单但有临界问题
  - 滑动窗口算法：解决临界问题
  - 漏桶算法：恒定速率
  - 令牌桶算法：允许突发流量

- **限流维度**：QPS、并发数、连接数、返回值

#### 3.5 服务降级
- **降级策略**：
  - 拒绝部分请求
  - 返回兜底数据/默认值
  - 返回友好提示（如"系统繁忙"）
- **降级开关**：可通过配置中心动态控制

#### 3.6 服务熔断
- **熔断器模式**（Circuit Breaker）：
  - **Closed（关闭）**：正常状态，统计失败次数
  - **Open（打开）**：快速失败，不再访问后端
  - **Half-Open（半开）**：试探恢复，允许部分请求通过

- 熔断阈值：失败次数/比例、超时时间

## 重要知识点

1. **服务发现解决了什么问题**：分布式环境中服务实例地址动态变化的问题

2. **负载均衡的核心算法**：
   - 一致性哈希：减少重新映射，适合分布式缓存
   - 令牌桶：允许一定程度的突发流量

3. **熔断 vs 降级**：
   - 熔断：保护**调用方**，避免雪崩
   - 降级：保护**被调用方**，提供兜底

## 本章小结
第2-3章介绍了分布式系统中最基础也是最核心的通信与协作机制。服务发现、负载均衡、限流熔断等概念是构建高可用分布式系统的基础组件，必须深入理解。
