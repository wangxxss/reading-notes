# 第11-13章：分布式数据与事务

## 本章要点

### 第11章：分布式数据管理

#### 11.1 分布式数据存储架构

##### 主从复制
```
         ┌──────────┐
         │  写入   │
         └────┬─────┘
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
┌────────┐ ┌────────┐ ┌────────┐
│ Master │ │ Slave1 │ │ Slave2 │
│ 主库   │ │ 从库1  │ │ 从库2  │
└────────┘ └────────┘ └────────┘
              │         │
              └────┬────┘
                   ▼
               ┌────────┐
               │  读取  │
               └────────┘
```
- 优点：读写分离，提高读取性能
- 缺点：主从延迟、数据一致性问题

##### 主主复制
- 两个节点都可以写入
- 优点：写入可扩展
- 缺点：冲突解决复杂

##### 分片（Sharding）
- 数据水平拆分到多个节点
- 分片键选择至关重要
- 常见分片策略：范围分片、哈希分片

#### 11.2 分布式ID生成方案

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **UUID** | 128位随机数 | 无需中心节点、性能高 | 无序、存储空间大、不利于索引 |
| **数据库自增** | auto_increment | 简单有序 | 单点瓶颈、扩展性差 |
| **号段模式** | 批量获取ID段，本地分配 | 性能高、可用性好 | 需要数据库支持 |
| **雪花算法（Snowflake）** | 时间戳+机器ID+序列号 | 有序、性能高、去中心化 | 时钟回拨问题 |
| **Leaf（美团）** | 号段模式+雪花算法双模式 | 生产验证、功能完善 | 部署复杂度 |

##### 雪花算法结构（64位）

```
0 | 41位时间戳 | 10位机器ID | 12位序列号
  | (约69年)   | (1024节点) | (每ms 4096个)
```

##### 雪花算法的时钟回拨问题与解决方案

时钟回拨是指服务器时间被回拨到过去，导致生成的ID可能重复或小于之前生成的ID。

**解决方案**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **等待回拨过去** | 检测到回拨时，等待时间追上来 | 简单 | 可能阻塞业务调用 |
| **历史最大值** | 记录上次生成的最大时间戳，下次从max(当前时间,历史最大值)开始 | 无需等待 | 依赖持久化存储 |
| **扩展位数** | 借用序列号位数应对时钟回拨（如增大时间戳位数减少序列号） | 灵活性高 | 调整后无法兼容旧ID |
| **双odus** | 预生成两个ID池（正常池、回拨池），回拨时使用回拨池 | 性能好 | 实现复杂 |

美团Leaf的解决方案：每个Leaf终端启动时读取ZK的时间戳最大值，并定期同步。如果发生时钟回拨，Leaf会借用sequence号段的上限来避免ID冲突。

##### UUID版本对比

| 版本 | 生成方式 | 有序性 | 隐私安全 | 适用场景 |
|------|---------|--------|---------|---------|
| **UUID v1** | 时间戳 + 节点MAC地址 | 趋势递增 | **不安全**（暴露MAC地址） | 不推荐 |
| **UUID v4** | 随机数 | 无序 | 安全 | 安全性要求高的场景 |
| **UUID v7** | 时间戳 + 随机数 | **递增**（Unix Epoch毫秒） | 安全 | 需要有序ID的场景 |

**推荐**：现代应用优先使用UUID v7（有序且安全），或雪花算法（高并发场景）。

##### 分布式ID选型建议

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 数据库主键（MySQL） | 雪花算法 / UUID v7 | 有序利于索引 |
| 分布式锁唯一键 | UUID v4 | 无需中心，高安全 |
| 消息队列消息ID | 雪花算法 | 高吞吐，有序 |
| 用户Session ID | UUID v4 | 高安全，无业务含义 |
| 订单号 | 雪花算法 + 业务前缀 | 可追溯，高性能 |

#### 11.3 数据分片

##### 分片策略
| 策略 | 优点 | 缺点 |
|------|------|------|
| 范围分片 | 查询友好 | 数据不均匀 |
| 哈希分片 | 数据均匀 | 查询需要广播 |
| 一致性哈希 | 扩展性好 | 实现复杂 |

- **ShardingSphere**：Apache开源，分布式数据库中间件
- **MyCAT**：基于MySQL的数据库中间件
- **Cobar**：阿里开源（已停止维护）

#### 11.4 分布式缓存

##### 缓存架构模式
```
┌─────────────┐         ┌─────────────┐
│   Client    │         │   Client   │
└──────┬──────┘         └──────┬──────┘
       │                       │
       ▼                       ▼
┌─────────────┐         ┌─────────────┐
│   Gateway   │         │   Gateway   │
└──────┬──────┘         └──────┬──────┘
       │                       │
       ▼                       ▼
┌─────────────┐         ┌─────────────┐
│   Service   │         │   Service   │
└──────┬──────┘         └──────┬──────┘
       │                       │
       ▼                       ▼
┌─────────────┐         ┌─────────────┐
│   Cache     │◄────────►│   Cache    │
│  (Redis)   │  同步    │  (Redis)   │

```

##### 缓存问题
| 问题 | 说明 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据 | 布隆过滤器、缓存空值 |
| **缓存击穿** | 热点key失效瞬间大量请求 | 互斥锁、永不过期 |
| **缓存雪崩** | 大量缓存同时失效 | 过期时间加随机值 |

### 第12章：分布式事务

#### 12.1 本地事务 vs 分布式事务

##### 本地事务
- 单个数据库节点内的事务
- 使用ACID保证一致性
- 无法跨多个数据库节点

##### 分布式事务
- 跨多个数据库/服务的事务
- 需要在分布式环境中保证ACID
- 面临CAP约束

#### 12.2 理论基础
> CAP定理与BASE理论的详细内容见 [第6-7章：容错与高可用](第6-7章_容错与高可用.md)。
>
> 核心要点：分布式事务方案的选择本质上是在**一致性**与**可用性/性能**之间的权衡。强一致方案（2PC/3PC）牺牲性能，最终一致方案（SAGA/消息表）牺牲实时性。

#### 12.3 分布式事务解决方案

##### 2PC（Two Phase Commit）
```
Phase 1: 准备阶段
协调者 ──── 准备请求 ────▶ 所有参与者
协调者 ◀─── 准备成功 ──── 所有参与者

Phase 2: 提交阶段
协调者 ──── 提交请求 ────▶ 所有参与者
协调者 ◀─── 提交成功 ──── 所有参与者
```

- **优点**：强一致性
- **缺点**：同步阻塞、单点故障、数据不一致风险
- **使用场景**：对一致性要求高的场景，不适合高并发

##### 3PC（Three Phase Commit）
- 在2PC基础上增加：
  - CanCommit阶段：检查是否可以提交
  - 参与者超时机制
- 减少阻塞时间，降低数据不一致风险

##### TCC（Try-Confirm-Cancel）
```
Try: 预留资源
     ┌─────────────┐
     │  冻结库存   │
     │ (库存-预留) │
     └─────────────┘

Confirm: 确认执行
     ┌─────────────┐
     │  扣除库存   │
     │ (冻结->已用)│
     └─────────────┘

Cancel: 取消回滚
     ┌─────────────┐
     │  释放库存   │
     │ (库存恢复) │
     └─────────────┘
```

- **优点**：性能较好，不锁定资源
- **缺点**：业务侵入性大，需要实现三个接口
- **使用场景**：性能要求高的场景

##### SAGA模式
- 将长事务拆分为多个子事务
- 每个子事务有对应的补偿事务
- 正向执行，失败则逆向补偿

```
A → B → C → D → E
    ↓   ↓   ↓
   补偿 补偿 补偿
```

- **类型**：
  - 编排式SAGA：中心节点协调
  - 协作式SAGA：各服务自协调
- **缺点**：补偿逻辑复杂，可能出现脑裂

##### 本地消息表
- 将消息写入本地数据库
- 可靠消息投递保证
- 事务与消息解耦

```
订单服务 ──写订单──▶ 本地表 ──投递──▶ 库存服务
     │                               │
     │◀──确认消息──│◀──确认消息────│
```

##### 可靠消息队列方案
- 利用消息队列的**事务消息**功能保证消息投递的可靠性
- **RocketMQ 事务消息**：
  1. 发送半消息（Half Message）到 MQ
  2. 执行本地事务
  3. 根据本地事务结果提交或回滚消息
  4. MQ 定时回查本地事务状态（防止异常情况）
- **Kafka Exactly-Once**：
  - 幂等生产者（idempotent producer）+ 事务 API
  - 保证消息不重复、不丢失
- **适用场景**：异步解耦的跨服务数据一致性

##### 最大努力通知
- 尽最大努力通知目标服务
- 适用于不重要的跨系统通知
- 通过定期核对保证最终一致

### 第13章：一致性算法

#### 13.1 一致性分类
| 类型 | 说明 | 场景 |
|------|------|------|
| **强一致性** | 更新后所有副本立即可见 | 金融、库存 |
| **弱一致性** | 不保证立即可见 | 缓存、CDN |
| **最终一致性** | 更新后最终所有副本一致 | 社交Feed、日志 |

#### 13.2 一致性协议

##### Paxos
- Lesley Lamport 提出的共识算法
- **角色**：Proposer（提议者）、Acceptor（接受者）、Learner（学习者）
- **达成共识的条件**：多数派（n/2+1）接受
- **两种变体**：Multi-Paxos、Fast-Paxos

**Paxos算法流程**：
```
Phase 1: Prepare
Proposer: 发起提议(n)，请求Promise
Acceptor: 如果n>已接受的提议，返回Promise

Phase 2: Accept
Proposer: 收到多数派Promise，发送Accept(n, value)
Acceptor: 如果n是最高编号，接受并广播
```

- **优点**：理论完善，活性保证
- **缺点**：难以理解，实现复杂
- **注意**：Zookeeper 使用的是 **ZAB 协议**（Zookeeper Atomic Broadcast），基于 Paxos 思想但有所不同——ZAB 是主从架构，强调全序广播；经典 Paxos 是对等架构

##### Raft
- Paxos的简化实现，更易理解和实现
- **角色**：Leader、Follower、Candidate
- **Leader选举**：term递增，获得多数票当选

**Raft子问题**：
1. Leader选举
2. 日志复制
3. 成员变更

```
        ┌─────────────────────────────────────────┐
        │               Raft 状态机               │
        ├─────────────────────────────────────────┤
        │                                         │
        │    ┌─────┐                              │
        │    │Follower│◀─── election timeout ───┐ │
        │    └─────┘                            │ │
        │        │                              │ │
        │        │ 收到Leader心跳                 │ │
        │        ▼                              │ │
        │    ┌─────┐     赢得选举       ┌─────┐ │ │
        │    │Candidate│ ─────────────▶│Leader│─┘ │
        │    └─────┘                   └─────┘   │
        │        │                              │
        │        │ 其他Candidate赢得选举        │
        │        ▼                              │
        │    ┌─────┐                            │
        │    │Follower│◀────────────────────────┘
        │    └─────┘
        └─────────────────────────────────────────┘
```

##### Gossip协议
- 去中心化的最终一致性协议
- 适合大规模分布式系统
- **传播方式**：随机选择节点，互相传播

```
节点A: v1 ──传播──▶ 节点B: v1
                   │
                   ▼
              节点C: v1 ──传播──▶ 节点D: v1
                   ...
最终所有节点都达到v1（最终一致性）
```

- **特点**：
  - 最终一致性，不是强一致性
  - 支持节点动态加入/离开
  - 网络开销随时间递减

### 一致性算法对比

| 算法 | 中心化 | 一致性 | 性能 | 代表系统 |
|------|--------|--------|------|---------|
| **Paxos** | 否 | 强一致 | 中 | Chubby（Google） |
| **Raft** | 是 | 强一致 | 中 | etcd、Consul |
| **Gossip** | 否 | 最终一致 | 高 | Cassandra、DynamoDB |

## 本章小结
分布式数据管理和事务是分布式系统中最复杂的话题之一。没有银弹，需要根据业务场景在一致性、可用性、性能之间做权衡。理解各种一致性算法的原理和适用场景，是架构设计的基础。
