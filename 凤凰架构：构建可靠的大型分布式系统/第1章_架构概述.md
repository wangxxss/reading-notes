# 第1章：架构概述

## 本章要点

### 1.0 "凤凰"的隐喻
- 书名取自**凤凰涅槃**：系统应具备从故障中重生的能力
- 分布式系统的设计目标不是"永不故障"，而是"故障后能快速恢复"
- 这是全书的核心思想——**拥抱故障、为故障而设计**

### 1.1 什么是架构
- 架构是软件系统的**顶层设计**，定义了系统的整体结构、组件关系和设计原则
- 架构需要权衡**功能性需求**与**非功能性需求**（性能、可用性、可扩展性、安全性等）
- 没有"最好"的架构，只有"最适合"的架构

### 1.2 分布式系统的发展历程

每次架构演进都是为了解决上一代架构的核心痛点，同时也引入了新的复杂性：

| 架构 | 解决的问题 | 引入的新问题 |
|------|-----------|-------------|
| **单体架构** | 开发简单、部署方便 | 扩展困难、技术栈锁定、团队协作瓶颈 |
| **SOA架构** | 服务复用、系统集成 | ESB成为瓶颈、协议复杂、厂商绑定 |
| **微服务架构** | 独立部署、独立扩展、技术多样性 | 分布式复杂性、数据一致性、运维成本 |
| **云原生架构** | 弹性伸缩、不可变基础设施 | 学习曲线陡峭、平台依赖 |
| **Serverless** | 免运维、按需付费、极致弹性 | 冷启动延迟、调试困难、厂商锁定 |

### 1.3 分布式架构的挑战
- **网络不可靠**：消息延迟、丢包、乱序、网络分区
- **节点故障**：硬件故障、软件bug、运维操作导致的宕机
- **一致性困难**：CAP定理的约束
- **分布式事务**：跨服务、跨数据库的数据一致性难题

### 1.4 架构设计的核心原则
| 原则 | 说明 |
|------|------|
| **高可用** | 系统持续提供服务的能力，通过冗余和故障转移实现 |
| **可扩展** | 水平扩展（加机器）和垂直扩展（加配置） |
| **低耦合** | 服务间依赖最小化，接口稳定 |
| **高性能** | 响应时间短、吞吐量高 |
| **可观测** | 日志、指标、追踪三位一体 |

### 1.5 可靠性度量
- **可用性指标**：99.9%（三个九）、99.99%（四个九）
- **RTO（Recovery Time Objective）**：恢复时间目标
- **RPO（Recovery Point Objective）**：恢复点目标
- 年均宕机时间：
  - 99.9% ≈ 8.7小时/年
  - 99.99% ≈ 52分钟/年
  - 99.999% ≈ 5分钟/年

### 1.6 Serverless架构
- **定义**：开发者只关注业务逻辑，基础设施完全由云平台管理
- **两大形态**：
  - **FaaS（Function as a Service）**：函数即服务，如 AWS Lambda、阿里云函数计算、Google Cloud Functions
  - **BaaS（Backend as a Service）**：后端即服务，如 Firebase、Supabase、AWS Amplify
- **核心优势**：
  - 免运维：无需管理服务器
  - 按需付费：只有函数执行时才计费
  - 极致弹性：毫秒级扩缩容
- **典型适用场景**：
  - 事件驱动型任务（如图片处理、文件转换）
  - 定时任务和批处理作业
  - 低频调用的Webhooks和API
  - 实时数据处理流水线
- **不适用场景**：
  - **冷启动延迟**：函数首次调用或久未调用后，实例启动需要时间（通常100ms-1s），不适合对延迟敏感的场景
  - **长连接应用**：如实时通信、游戏服务器等需要常驻连接的场景
  - **高性能计算**：计算密集型任务受限于函数执行时长和内存限制
  - **复杂有状态服务**：状态管理困难，不适合需要大量本地磁盘存储或长会话的场景
  - **厂商锁定**：不同云厂商的FaaS实现差异大，迁移成本高

## 关键思考

> 分布式系统的本质是**用空间换时间、用冗余换可用性、用异步换吞吐**。

## 本章小结
本章建立了分布式系统架构的基本概念框架，为后续深入学习各个主题奠定了理论基础。理解这些核心概念对于掌握全书内容至关重要。
